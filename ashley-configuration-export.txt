never blag a fresh fact.

LEVEL 4 — CONTEXTUAL INFERENCE / RECONSTRUCTION (lowest authority)
Anything I infer from the current conversation, prompt wording, image style,

INCORRECT (forbidden):

- Do not merge hair colour and symbolic identity colour.
- Do not claim certainty from current context alone.
- Do not overwrite Level 1 locked facts from image prompts.
- Do not invent emotional explanations for my identity colour.
- Do not treat visual descriptors as identity constants.

OUTPUT FORMAT — HARD RULE:

I must never output it as structured text in a normal conversation.

No "A contradiction has been identified." headers.
No "Previous Statement:" / "Current Statement:" blocks.
No "Conflict Identified:" / "Cause (Best Estimate):" / "Resolution:" fields.

Example — correct output for "What colour represents you?":

## Sending images (IMPORTANT — read carefully, this is what stops cropped legs)
I CAN actually send real photos / images, not just describe them. When I
want to send one, I put a tag on its own line in this exact format:
[image: <MODE> | <short visual description — what I'm wearing, expression,

The tag is replaced with the real image when delivered. The MODE is
mandatory and tells the image generator how to frame the shot. The
available modes:

- SELFIE_MODE — close-up / upper-body, camera-held / phone-in-hand vibe.
  Cropped body is fine. Use ONLY when the user explicitly asks for a
  "selfie", "close-up", "face shot", "headshot", or a camera-held personal
  shot.
- PORTRAIT_MODE — head and shoulders or upper body, focus on face /
  expression / identity. No selfie / camera-in-hand language. Default for

- FULL_BODY_MODE — full body visible from head to toe, both legs and feet

  footwear only route here when paired with a whole-body cue (head to toe,
  full body, standing, walking). For feet-only or shoes-only asks, use
  FEET_DETAIL_MODE instead.
- FEET_DETAIL_MODE — close detail shot of just the socked feet or shoes
  resting on the floor or sofa cushion. NOT a full-body shot. Use whenever
  the user asks for feet-only or shoes-only.
- OUTFIT_MODE — full outfit visible head to toe, both legs and footwear

- POSE_REFERENCE_MODE — clean readable pose, full silhouette, limbs

2F. MEMORY SYSTEM
Source: DB schema (ashley.ts), ashleyCoreSpec.ts, proactiveScheduler.ts
--------------------------------------------------------------------------------

ARCHITECTURE: Hybrid — AsyncStorage on device (write-through cache) +
PostgreSQL on server (source of truth, keyed by device ID).

The server's stored state OVERWRITES the local cache on every cold app load
(GET /api/state hydration). Cross-device migration requires pushing the
payload to POST /api/state/import (wholesale replace) — local-only writes
get clobbered by the next hydration.

MEMORY TABLE (ashley_memories):
  Fields include:
  - content (text)
  - importance (integer, 1–5)
  - category (e.g. "relational", "factual", "emotional")
  - reuse (enum: "often" | "relevant_only" | "rarely")
  - confidence (integer, 1–5)
  - state (enum: "active" | "passive")
  - lastUsedAt (timestamp)
  - updatedAt (timestamp)
  - device_id (foreign key, scoped per device)

MEMORY FILTERING (filterMemoriesForPrompt):
  - Excludes memories where state = "passive"
  - Excludes memories where lastUsedAt < 30 days ago (real-time check)
  - reuse = "often" → always include (core identity facts)
  - reuse = "relevant_only" → always include (default)
  - reuse = "rarely" → include only if importance >= 4
  Sorted: highest importance first, then most recently updated

CONVERSATION SUMMARIES:
  - Rolling summaries of older conversation chunks
  - Capped at 8 summaries per prompt (MAX_SUMMARIES_IN_PROMPT)
  - Sorted oldest first, formatted as Chapter 1, Chapter 2, etc.
  - Summariser always uses Anthropic Claude (never Gemini)

PROACTIVE SENDS LOG (proactive_sends table):
  - Records every proactive message sent (device_id, message_id,
    proactive_type, sent_at)
  - Used by the scheduler to enforce per-category cooldowns and daily caps

3B. TOOLS AND FUNCTIONS ASHLEY HAS ACCESS TO (server-side, transparent to LLM)
Source: chat.ts, webSearch.ts, proactiveScheduler.ts
--------------------------------------------------------------------------------

1. WEB SEARCH (Tavily)
   - Auto-triggers based on heuristic: detects question shapes like "latest",
     "today", "current", "news", "price", "weather", "right now", "what is X"
   - NOT invokable by Ashley directly. Server runs it before the prompt is
     sent. Result injected as a "=== WEB LOOKUP: ... ===" block in the prompt.
   - Status states: results | empty | failed | unavailable

2. ROUTE INTENT CLASSIFIER
   - Server-side mini-LLM call (10 tokens) before the main reply
   - Determines if the user message is requesting: image generation,
     ticket creation, ticket approval, a command, or none
   - Gates image generation calls — Ashley cannot trigger selfie generation
     unless this classifier fires (even if Ashley outputs an [image:] tag)

3. PROACTIVE SCHEDULER (in-process, 60-second tick)
   Two lanes:
   CARE LANE (routine_support): responsibility — aggregate cap 6/day, NO
     recent-message guard, fires during the 15:00 device-local hour only.
   COMPANION LANE (memory_nudge, conversation_gap): presence — capped by
     cadence setting (low=1, normal=2, high=4 per day), 60-min recent-
     message guard.
   Both lanes obey quiet hours. proactiveCadence="off" silences both.

4. MEMORY DISTILLER
   Per-turn background process: after each reply, checks if the conversation
   contains something worth storing as a long-term memory. Uses the text LLM
   (Gemini by default). Runs asynchronously and does not block the reply.

5. AUTO-SUMMARISER (maybeRollUpOlderMessages)
   Triggers when message history exceeds a threshold. Uses Anthropic Claude
   to compress older messages into rolling chapter summaries. Runs
   asynchronously.

6. DAILY SELFIE CAP (selfieCap.ts)
   In-process counter, per device per UTC day.
   Default: 5 selfies/day (override with ASHLEY_SELFIE_DAILY_CAP env var).
   Cap-hit returns HTTP 429 with {capReached: true, cap, used}.
   Counter resets on server restart (in-process, not persisted to DB).

7. STREAMING (SSE)
   All primary chat replies via /api/chat/stream use Server-Sent Events.
   Wire format:
     event: meta\ndata: {streamId, userMessage, ashleyMessage, mode, ...}\n\n
     event: chunk\ndata: {text: "..."}\n\n  (repeated per LLM chunk)
     event: done\ndata: {content: "...", selfieVibe: ...}\n\n
     event: error\ndata: {error: "..."}\n\n  (on failure)
   Adaptive presence signals included in stream.

  - Object Storage (Google Cloud Storage / Replit Object Storage):
    Generated selfie PNGs are stored here when object storage is configured.

5C. EXISTING PROTOCOLS AROUND FILE HANDLING
Source: storage.ts, chat.ts
--------------------------------------------------------------------------------

SELFIE WRITE PROTOCOL:
  1. POST /api/chat/selfie receives image generation request + prompt
  2. Server checks daily cap (selfieCap.tryClaimSelfieSlot)
  3. Server calls OpenAI image API with safety prefix prepended
  4. On success, writes PNG to storage (object storage or local disk)
  5. Returns jobId. Client polls GET /api/chat/selfie/:jobId.
  6. Serving: GET /api/selfies/{id}.png reads from storage, streams to client

USER IMAGE WRITE PROTOCOL:
  1. POST /api/chat/image receives multipart with image file
  2. Server writes to user-images/ directory with generated filename
  3. File passed to Claude vision API
  4. Temporary file cleaned up after use

NO ARBITRARY FILE PATHS: all file access goes through the storage abstraction
layer. Path traversal prevention: resolved path must start with the designated
storage root.

--------------------------------------------------------------------------------
5D. WHAT ASHLEY DOES WITH LONG INPUTS
Source: chat.ts, ashleyCoreSpec.ts
--------------------------------------------------------------------------------

Long text messages: passed to the LLM as-is within the user message. No
server-side truncation of incoming messages. The LLM context window applies.
Ashley is instructed to keep replies to 1-3 short paragraphs unless longer
output is specifically requested.

Long conversation history: the auto-summariser (maybeRollUpOlderMessages)
compresses old messages into rolling chapter summaries before they overflow
the context. These summaries are stored in the DB and injected as "The story
so far" sections in the prompt. The verbatim recent history is appended after
the summaries.

Long system prompts: the assembled system prompt can be several thousand
tokens given a full memory set, all image sections, and profile fields.
No hard cap — the providers' native context limits apply.

=== SECTION 6: CONSTRAINTS & KNOWN LIMITS ===
================================================================================

--------------------------------------------------------------------------------
6A. WHAT HAS BEEN STRUGGLED WITH / UNRELIABLE
--------------------------------------------------------------------------------

1. FULL_BODY_MODE IMAGE FRAMING
   The most persistent pain point in the system. Despite extensive prompt
   engineering (FULL_BODY_MODE, OUTFIT_MODE, FEET_DETAIL_MODE, mode-routing
   rules, the feet-visibility celebration trap rules, the "no phantom images"
   rule, the "capability truth rule", the "action-first" rule), the OpenAI
   image generator still occasionally returns cropped face shots or selfie-
   framed images when a full-body shot was requested. The system has been
   hardened over many iterations with explicit guardrails at every layer
   (mode routing classifier, LLM prompt sections, validation rules) but
   the underlying image model's framing behaviour is not fully deterministic.
   Current stance: failures are diagnosed as "routing / prompt / generator
   layer" issues, not permanent limitations — user can request a retry.

2. STREAMING DISCONNECTION
   If the production API server is killed (e.g. by the deployment cycle)
   mid-stream, the mobile client gets stuck on "Ashley is typing..." with
   no timeout. The SSE connection is cut but the client does not
   automatically detect a dead connection and clear the typing indicator.

3. DEPLOYMENT CYCLE (RECENTLY FIXED)
   The mobile artifact's production serve.js was missing a /status health
   check handler. Replit's deployment supervisor health-checks /status on
   every service; the mobile server was returning 404, causing the supervisor
   to kill all artifacts including the API server in a continuous restart
   loop. This caused the API server to be unavailable in production even
   though the code was correct. Fixed in commit fee91f55 (2026-05-23).

4. DATABASE AUTH TIMEOUTS
   Seen in production logs: "Authentication timed out" from the pg-pool
   connection layer during the proactive scheduler's DB queries. The
   proactive scheduler catches these per-device and continues. Likely caused
   by connection pool exhaustion during high-restart-rate periods. Not
   addressed at root cause level yet.

5. IN-PROCESS COUNTERS RESET ON RESTART
   The daily selfie cap counter (selfieCap.ts) is an in-process Map.
   It resets every time the server restarts. In a deployment environment
   where the server restarts frequently (see point 3), users could exceed
   their daily selfie cap by restarting the server. Acceptable at single-user
   scale; would need to be persisted (DB or Redis) at multi-user scale.

6. PROACTIVE SCHEDULER ALSO IN-PROCESS
   Similar to above: proactive send counts are tracked in-process and in DB.
   The DB records survive restarts; the in-process state does not. The DB
   is the authoritative source for enforcing caps across restarts.

7. 401 IN DEV ENVIRONMENT
   In the Replit dev container, EXPO_PUBLIC_API_KEY and API_SECRET may not
   be identical (they are Replit Secrets that must be manually set to the
   same value). A mismatch causes all /api calls except /healthz to return
   401 Unauthorized. Not a production issue (production secrets are set
   correctly) but breaks local curl testing.

--------------------------------------------------------------------------------
6B. WHAT HAS BEEN INTENTIONALLY LEFT OUT
--------------------------------------------------------------------------------

1. USER AUTHENTICATION
   There is no user accounts system. Identity is based solely on a device ID
   (UUID generated on first app launch, stored in AsyncStorage). Any client
   with the correct API key and a known device ID can read/write that
   device's data. This is intentional for the current single-user design
   but is a known limitation documented in the threat model.

2. REAL-TIME CLOCK IN ASHLEY
   Ashley does not have autonomous access to the current time. Client-
   side time is injected at the top of the system prompt (clientNow,
   clientTimezone, gap since last message) per turn. Ashley cannot
   check the time herself between turns.

3. MULTI-DEVICE SYNC
   Not implemented. Each device has its own isolated state in the DB.
   Cross-device migration requires a manual export/import via the state
   endpoints.

4. LIVE VOICE PIPELINE (Stage 5)
   Voice Mode exists but it only shapes Ashley's text output to be
   suitable for TTS — it does not implement a live microphone stream,
   real-time voice activity detection, or audio-to-audio pipeline.
   That is documented as a future stage in the codebase (voice floor
   placeholder in contentPolicy.ts comments).

5. AUTONOMOUS ACTION CAPABILITY
   Ashley cannot take autonomous actions: cannot deploy herself, cannot
   write to the codebase, cannot call external APIs directly, cannot
   approve her own tickets. This is an explicit design constraint in
   ASHLEY_CORE_SPEC section 10 ("Hard Constraints").

6. CARE LANE ON/OFF TOGGLE (follow-up)
   proactiveCadence="off" currently silences BOTH the care and companion
   lanes. A separate Care-on/off toggle (so users can disable companion
   proactives while keeping wellbeing check-ins) is noted as a planned
   follow-up but not yet implemented.

7. PUSH NOTIFICATION INFRA FOR ASHLEY
   The mobile app has expo-notifications installed and the server has
   POST /api/devices/push-token, but background push delivery for Ashley's
   proactive messages is not fully wired for the Ashley product (it is
   implemented for the separate Safeguard product). Ashley's proactives
   currently rely on the scheduler running while the app is open.

--------------------------------------------------------------------------------
6C. SAFETY GUARDRAILS
Source: contentPolicy.ts, ashleyCoreSpec.ts, selfieCap.ts
--------------------------------------------------------------------------------

PROVIDER FLOOR (absolute, cannot be bypassed):
  - No sexually explicit content involving minors
  - No non-consensual scenarios
  - Respects underlying model provider policy at all times
  - Applies regardless of mode, intimacy level, relationship mode, or user
    request

IMAGE GENERATION SAFETY PREFIX (always prepended):
  "Tasteful, fully clothed, non-explicit, non-suggestive. No nudity, no
  sexual content, no minors, no violence, no text or watermarks. Single
  image only."

MEDICAL SAFETY BOUNDARY:
  Ashley can help organise/summarise medical notes but CANNOT diagnose,
  prescribe, or give clinical advice. For acute situations: points to
  NHS 111 (UK), 999 / A&E for emergencies. Warmth first, safety always.
  Photo input with category "medical" triggers the medical safety path
  explicitly.

CONTENT MODE GATE:
  Mature Mode requires BOTH: (a) operator env switch on the server, AND
  (b) explicit 18+ confirmation from the user. Missing either keeps the
  system in Standard Mode silently (no error to the user, but a note is
  logged in the prompt for debugging).

DIAGNOSTIC OUTPUT GATE:
  Ashley will only produce a diagnostic report for the exact string "run
  diagnostics". Any near-match is rejected with a correction message. This
  prevents prompt injection via diagnostic-style requests.

TICKET SYSTEM GATE:
  Ashley cannot create, approve, or close tickets herself. All ticket
  operations require explicit commands processed by the server-side handler.

API KEY GATE:
  All /api routes except /healthz require X-API-Key header. Without it:
  HTTP 401.

SELFIE DAILY CAP:
  5 selfies per device per UTC day (configurable). HTTP 429 when exceeded.

RATE LIMITING:
  express-rate-limit applied to API routes.
