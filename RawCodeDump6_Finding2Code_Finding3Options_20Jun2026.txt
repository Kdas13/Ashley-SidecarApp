──────────────────────────────────────────────────────────────────────────────
IS THIS THE SAME AS THE PREVIOUS FREEZE DIAGNOSTIC (currentSpeechId gate)?

NO. The previous freeze was about the 45s MAX_TURN_BUDGET_MS safety net and
whether the currentSpeechId gate inside it would pass on a new turn. That
was a different turn, different path. This freeze:

  a) The 45s budget never fired — the error happened at T+8.2s and the
     interrupt was received at T+9.8s. The 45s timer was cancelled in the
     finally block (line 644: clearTimeout(hardBudget)).
  b) The freeze is caused by the 15s SAFETY TIMEOUT for playback_confirmed,
     not the 45s turn budget.
  c) The 15s timeout fires but its guard uses the same "currentSpeechId ===
     speechId" pattern — and the interrupt has cleared currentSpeechId to
     null by then, so the guard silently fails every time.
  d) The specific compound failure is: LLM error → error clip sent → 
     awaitingPlaybackConfirm=true → interrupt → currentSpeechId=null → 
     safety timeout becomes permanently ineffective → gate never clears.

──────────────────────────────────────────────────────────────────────────────
DOES THE ERROR PATH SEND ANYTHING TO THE CLIENT?

YES — it sends more than you might expect:
  1. tts_done (via speakClipOrText, for the error clip)
  2. response_end (after speakClipOrText returns, if currentSpeechId still matches)
  3. awaitingPlaybackConfirm = true (internal)

The client DOES receive both tts_done and response_end for the error path.
The client should theoretically send playback_confirmed in response. Why didn't it?

Likely: at T+9.8s the interrupt_ack arrived on the client. The client's interrupt_ack
handler (useVoiceCall.ts line 694–699) sets ttsCompleteRef=true, calls stopPlayback(),
sets phase to "listening", and calls openMicRef.current() — it moves on. Once the
client is in "listening" state with ttsCompleteRef=true, subsequent response_end
messages from the server (which the client receives after the interrupt_ack) would
reach a client that is already past the playback phase and may not trigger a
playback_confirmed send. The client's playback_confirmed logic is inside playNext
(called when audio playback ends) — if stopPlayback() was called by the interrupt,
the playback chain is broken and playback_confirmed is never sent, even though
response_end was received.

Both client and server are in an irrecoverable mismatched state:
  Server: awaitingPlaybackConfirm=true, waiting
  Client: in "listening" state, has moved past the turn, will not send playback_confirmed

FINDING 3 — THREE OPTIONS FOR PREVENTING DUPLICATE openMic() ACTIVATIONS
================================================================================

CONTEXT: two independent call sites both successfully invoke openMic() within a
~230ms window at a turn boundary. The mutex (micOpeningRef) works as designed —
it only blocks concurrent calls; the second call arrives after the first has
already completed and reset the mutex. Two VAD delay timers + two autoSubmit
timers run simultaneously until one fires and the other fires 10–20ms later.
Today this is log noise; it could cause a double-submit if a segment happens to
be ready exactly at the double-fire.

──────────────────────────────────────────────────────────────────────────────
OPTION A — Turn-boundary generation counter
──────────────────────────────────────────────────────────────────────────────

Add a `micGeneration` ref (integer, incremented each time openMic() is CALLED).
Each openMic() call captures the generation at entry. At the point where it would
arm timers and activate VAD (after recorder.start() succeeds), it checks whether
the captured generation is still current. A second call that started after the
first completed would see a different generation and be a no-op at the "activate"
point, even though it already started the recorder.

  const micGenRef = useRef(0);

  const openMic = useCallback(async () => {
    ...
    const gen = ++micGenRef.current;  // increment and capture

    await recorder.ensurePermission();
    await recorder.start();

    if (gen !== micGenRef.current) {
      // A newer openMic() has run since this one started.
      addLog("openMic: stale generation — suppressing");
      return;
    }
    // arm VAD delay, autoSubmit, log OK
  }, [...]);

TRADEOFFS:
  + Precise: the actual physical record() calls both run, but only the most
    recent generation activates VAD and arms timers.
  + No changes to the playback or submit paths.
  - Two recorder.start() calls still happen (they're fast and no-op on the
    second call if the recorder is already started, but it's not confirmed
    whether expo-av throws or ignores a second start while started).
  - If the recorder's start() has side effects on double-call, this doesn't
    protect against them.

──────────────────────────────────────────────────────────────────────────────
OPTION B — Identify and consolidate the two call sites
──────────────────────────────────────────────────────────────────────────────

The two call sites at the turn boundary are:
  Line 249: safety-timeout path in playNext (queue empty + ttsServerDoneRef=true)
  Line 698: interrupt_ack WS handler (or tts_done WS handler, depending on the turn)

These are conceptually duplicate: both say "audio is done, open mic". The safety-
timeout path is a client-side heuristic that predates the server's tts_done/
response_end protocol. If the tts_done path (line 698 or the playback_confirmed
→ tts_done chain) reliably fires, the safety-timeout path at line 249 is
redundant at that turn boundary.

Consolidation approach: in playNext's safety-timeout branch (line 243–249),
before calling openMicRef.current(), check whether micOpeningRef.current is
already false AND whether we just completed openMic within the last Nms. If yes,
skip the second call. This is a lighter version of Option C (see below), applied
only to this specific branch.

Alternatively: audit whether the safety-timeout path (ttsServerDoneRef=true) and
the interrupt_ack / tts_done paths are actually both needed at every turn boundary,
and whether one can be made authoritative, making the other conditional.

TRADEOFFS:
  + Targets the actual site of the duplication, not a general mechanism.
  + Would require careful analysis of every scenario where both paths fire (not
    just interrupt turns — also clean turns, reconnect turns, etc.) to ensure
    no regression.
  - More code surgery than Options A or C.
  - Risk: if the authoritative path fails (e.g. tts_done from server is lost),
    the safety-timeout path is needed as a fallback. Removing it without a
    replacement fallback could cause mic to never open.

──────────────────────────────────────────────────────────────────────────────
OPTION C — Last-openMic-OK cooldown ref (blunter backstop)
──────────────────────────────────────────────────────────────────────────────

Add a `lastOpenMicOkAtRef` ref (timestamp). At the end of openMic() (after
recorder.start() succeeds), record Date.now(). At the top of openMic(), before
the mutex check, also check whether a successful open happened within the last
OPENMICOK_COOLDOWN_MS (e.g. 300ms). If so, log and return.

  const lastOpenMicOkAtRef = useRef(0);
  const OPENMICOK_COOLDOWN_MS = 300;

  if (Date.now() - lastOpenMicOkAtRef.current < OPENMICOK_COOLDOWN_MS) {
    addLog("openMic COOLDOWN — skipped");
    return;
  }
  // ... existing mutex check ...
  // ... after recorder.start() succeeds: ...
  lastOpenMicOkAtRef.current = Date.now();

TRADEOFFS:
  + Simplest implementation — one ref, one check, one write.
  + No structural changes to call sites.
  - 300ms is a guess. If a call site LEGITIMATELY needs to open the mic within
    300ms of a previous open (e.g. rapid reconnect after network drop), it would
    be silently blocked.
  - Does not address WHY two paths fired — only prevents the second from
    completing. Leaves the root cause in place.
  - The cooldown value is a magic number that may need tuning per-device.

──────────────────────────────────────────────────────────────────────────────
RECOMMENDATION (not a request, just a call):

Option A (generation counter) is the most semantically precise — it allows the
recorder to start (which is hardware and can be idempotent) but ensures only one
generation arms VAD and timers. Option C is lower-effort but leaves the root
cause in place. Option B is the right long-term fix but needs a full audit of all
paths before touching it.

The issue is not urgent (duplicate timers are noise today). It can wait for a
hardening round if preferred.
