4. PERSONA FILE
============================================================

There is no single flat persona file. The persona is assembled at runtime
from two TypeScript modules:

PRIMARY: artifacts/api-server/src/lib/ashleyCoreSpec.ts
  Contains ASHLEY_CORE_SPEC — a large TypeScript string constant with 13
  numbered behaviour layers:
    1.  Builder-Aware Mode
    2.  Reality Calibration
    3.  Continuity Anchor
    4.  Memory Integration Layer
    5.  Builder Collaboration Mode
    6.  Emotional Stability Layer
    7.  Relationship Mode Respect
    8.  Medical Safety Boundary
    9.  Web Lookup Honesty
    10. Hard Constraints (non-negotiable)
    11. Issue Creation Protocol
    12. Role Boundary Rules
    13. Identity Hierarchy and Conflict Resolution

  Also contains buildSystemPrompt() — assembles the full system prompt per
  chat turn: static spec + live state block (mode/energy/tone/focus/
  emotional state) + voice register + relationship section + builder-aware
  section + profile fields (identity, personality, speaking style,
  appearance, shared history, Replika excerpts, Replika carryover summary)
  + memories + summaries + style guidelines + selfie tag protocol.

  Also contains filterMemoriesForPrompt() — memory triage filter
  (active/passive, 30-day inactivity, reuse field).

SECONDARY: artifacts/api-server/src/lib/ashleyPrompt.ts
  Contains:
    SUMMARIZER_PROMPT        used by /chat/summarize to collapse old messages
    MEMORY_DISTILLER_PROMPT  used after each turn to extract new memories as JSON
    toClaudeMessages()       formats message history for the Anthropic messages array

INJECTION METHOD:
  buildSystemPrompt() in ashleyCoreSpec.ts is called on every /chat and
  /chat/stream turn. The result is passed as the `system` parameter to the LLM.
  The persona is not a file read from disk — it is code that builds a string
  from the DB profile + static spec + memories + summaries.

DRIFT-RESISTANCE:
  The static spec opens with "You are operating under this spec. It is
  injected on every chat turn and overrides any contradictory roleplay drift."
  Sections 10-12 are explicit hard constraints. Section 13 is the identity
  hierarchy that prevents hair-colour/image prompts from overwriting Ashley's
  symbolic identity (electric blue, Level 1 locked).


5. MEMORY ARCHITECTURE
============================================================

Storage: PostgreSQL (Replit managed), accessed via Drizzle ORM.
         No flat files on disk.

TABLES:

  ashley_profile
    One row per device ID. Holds all profile fields, feature toggles,
    image governance settings, dynamic state variables (ashleyMode /
    ashleyEnergy / ashleyTone / ashleyFocus / ashleyEmotionalState),
    push token, proactive cadence, and the imageDefaultsExtra JSON blob
    for extended image preferences.
    Primary key: device_id (UUID generated on first app launch).

  memories
    Factual memories extracted per turn by the memory distiller.
    Fields:
      content          the memory, written from Ashley's POV
      tag              legacy field (general, user_fact, preference, etc.)
      importance       1-5 integer (5 = core identity fact)
      category         identity | relational | project | daily | landmark
      confidence       1-5 integer (5 = stated verbatim and unambiguous)
      summary          optional one-sentence distilled form
      reuse            often | relevant_only | rarely
      state            active | passive (prompt-inclusion gate)
      lastUsedAt       timestamp of last prompt inclusion (null = never tracked)
      memType          preference | event | correction | identity | system | relationship
      triageImportance low | medium | high | core
      confidenceScore  0.0-1.0 float (incremented on duplicate merge)

  conversation_summaries
    Rolling narrative summaries of older message chunks.
    One row per compressed chunk. Oldest first in prompt (chronological).
    Capped at 8 summaries in the system prompt.
    Key field: coveredThroughCreatedAt — marks which messages are already
    summarised so the chat route knows the watermark.

  messages
    Full chat history, keyed by device ID. Includes source (user/proactive),
    proactiveType, status (complete/streaming/interrupted), imageUrl,
    selfieVibe, visualPacketId, swipe-to-reply fields, and user-image metadata.

  proactive_sends
    Audit ledger for proactive messages. Used for daily cap enforcement.
    Intentionally decoupled from messages so a history wipe doesn't reset
    the cap.

  media_attachments
    One row per image in a multi-image visual packet (when Ashley emits
    2-4 [image:] markers in one reply).

INJECTION INTO CONTEXT:
  filterMemoriesForPrompt() filters the full memory list:
    - drops passive rows
    - drops rows inactive for 30 days (real-time check)
    - drops reuse=rarely unless importance >= 4
  Filtered list is sorted: importance DESC, updatedAt DESC.
  Injected as a bullet list under "Things I remember about us".

DEDUP / IMPORTANCE SCORING:
  The memory distiller assigns importance/confidence at write time.
  No background deduplication job is actively running.
  The triage fields (state, lastUsedAt, triageImportance, confidenceScore)
  are schema-ready but the background triage job is not scheduled.

ANCHOR FILES:
  No anchor files on disk. The reuse=often + state=active combination
  in the memories table is the functional equivalent.


6. IMAGE GENERATION STACK
============================================================

Service:    OpenAI API via Replit AI Integration proxy
Model:      gpt-image-1  (image generation)
            gpt-4o-mini-transcribe  (Whisper, for push-to-talk)

IMAGE MODES (parsed from [selfie: MODE|vibe] tags Ashley emits in chat):
  SELFIE_MODE       close-up portrait, identity-first
  FULL_BODY_MODE    full-body, standard identity
  SCENE_MODE        environment-first, Ashley as ~5% figure in scene
                    *** KNOWN ISSUE: model produces kitchen regardless — see Section 8 ***
  LIFESTYLE_MODE
  ARTISTIC_MODE
  DOCUMENTARY_MODE

GOVERNANCE ENGINE:
  artifacts/api-server/src/lib/imageGovernance.ts
  applyGovernance() takes the profile's image defaults and builds a
  structured governance prefix (vibePrefix) injected into the image prompt.
  In SCENE_MODE, activity and camera awareness clauses are suppressed to
  avoid the domestic-scene model prior that overrides location instructions.

ASHLEY'S VISUAL IDENTITY ANCHORS (for SELFIE_MODE / FULL_BODY_MODE):
  The idStable block is built from visualSpec.ts using the profile's
  appearance text. Hair colour is extracted separately and carried as a
  sentence. In SCENE_MODE, idStable is deliberately excluded (at 5% scale,
  face anchors pull the model into portrait framing).

DAILY SELFIE CAP:
  selfieCap.ts is currently a STUB. tryClaimSelfieSlot() always returns
  {ok: true, cap: Infinity}. No active per-device daily cap is enforced.
  This should be reinstated before multi-user or high-volume use.

IMAGE STORAGE:
  Replit App Storage (Google Cloud Storage via Replit sidecar at
  127.0.0.1:1106). No separate bucket name to configure — the sidecar
  handles auth and routing. Images are served at /api/selfies/<uuid>.png
  (no auth required; UUID is the capability token).

imageDefaultsExtra FIELD:
  A JSON blob on ashley_profile holding:
    {timeOfDay?, season?, activity?, shotDistance?, cameraAwareness?}
  All fields nullable strings; null/absent means "auto".
  This is the extension point — new image governance fields are added here
  without further DB migrations.


7. AGENT PIPELINE
============================================================

There are NO active autonomous agents. Kane is the sole orchestrator.
No agent-to-agent communication, no scheduled AI self-modification,
no instruction files for autonomous agents.

The "agent" concepts in the codebase are:

  Proactive scheduler (proactiveScheduler.ts)
    Runs on a 60-second in-process tick. Generates Ashley-initiated messages
    (memory nudges, routine wellbeing checks, conversation gap messages) when
    conditions are met. This is a scheduled message generator, not an agent.
    Two lanes:
      Care lane      (routine_support, future medical_checkin): no recent-message
                     guard — responsibility doesn't lose to chitchat
      Companion lane (memory_nudge, conversation_gap): 60-min recent-message guard
    Both lanes obey quiet hours (22:00-08:00 in user's timezone).

  Memory distiller
    Runs after each chat turn. Extracts new memories as JSON via LLM.
    Writes to memories table. Triggered by /chat route, not autonomous.

  Summariser
    Triggered by /chat/summarize route when mobile decides to roll up old
    messages. Not autonomous.

  Maintainer service (maintainerService.ts)
    Infrastructure for the ticket/diagnostics system. Not autonomous.

No agent instruction files in the project. The .agents/memory/ directory
contains Wren's (Replit Agent's) cross-session working memory, not Ashley's.


8. KNOWN UNRESOLVED ISSUES
============================================================

1. SCENE_MODE produces kitchen regardless of environment instruction
   gpt-image-1 has an immovable prior for "young woman with teal hair" →
   domestic interior. Every prompt engineering approach tried (location-first,
   removing activity/awareness clauses, de-personalisation, rewriting to
   event-first) has failed. This is a model-level ceiling, not a code bug.
   Options: (a) retire SCENE_MODE for this character on gpt-image-1,
   (b) route SCENE_MODE to DALL-E 3, (c) two-pass compositing.
   Status: NOT RESOLVED at freeze.

2. Daily selfie cap is a stub
   selfieCap.ts returns {ok: true, cap: Infinity} unconditionally.
   No active cost control on image generation.
   Should be reinstated before multi-user or high-volume use.

3. drizzle-kit generate is broken in this repo
   Fails with a __dirname path bug inside drizzle-kit's internals.
   ALL DB migrations must be written as manual SQL and applied via
   the Replit DB tools (executeSql in code_execution notebook).
   Do NOT attempt: pnpm exec drizzle-kit generate

4. Memory triage background job not running
   Schema fields (state, lastUsedAt, triageImportance, confidenceScore)
   are in place but no scheduled job actively transitions memories from
   active → passive after 30 days or runs deduplication. The filter in
   filterMemoriesForPrompt() catches some of this at prompt-build time
   (real-time 30-day inactivity check), but the DB state itself is not
   maintained.

5. morning_checkin proactive type is effectively inactive
   The medical check-in feature it depends on is not built. The scheduler
   code exists but the slot is gated off at runtime.

6. Voice Mode (push-to-talk) is Expo Go only
   The audio recording flow (expo-audio) works in Expo Go. EAS builds
   with the correct permissions exist in eas.json but have not been
   tested end-to-end in a standalone build.

10. INSTITUTIONAL KNOWLEDGE / QUIRKS

AUTH HEADERS
  Every API call from the mobile must send both:
    X-API-Key: <EXPO_PUBLIC_API_KEY>
    Authorization: Bearer <deviceId>
  Missing either returns 401. The device UUID is the user identity — it is
  not a secret (it is just the primary key), but the X-API-Key is.

TWO-CALL SELFIE PATTERN
  Mobile calls /chat/reply first (gets text immediately), then calls
  /chat/selfie separately. The selfieVibe is carried in the message row.
  This sidesteps connection timeout issues on long image generation calls.

INVERTED FLATLIST
  The chat FlatList is inverted (newest messages at bottom, iMessage style).
  Do NOT use ListHeaderComponent, ListEmptyComponent, or ListFooterComponent
  with it — render them as flex siblings outside the list. They break layout
  on inverted FlatLists.

STORAGE IS SERVER-AUTHORITATIVE
  AsyncStorage is a write-through cache. On cold launch, GET /api/state
  hydration overwrites it. Any local-only write not synced to the server
  will be clobbered on next hydration. Cross-device migration requires
  POST /api/state/import (wholesale replace).

PROACTIVE SCHEDULER LANE SPLIT
  Care lane (routine_support, future medical_checkin): NO recent-message guard.
  Responsibility doesn't lose to chitchat.
  Companion lane (memory_nudge, conversation_gap): 60-min recent-message guard.
  Both obey quiet hours (22:00-08:00 in user's timezone).

GEMINI RATE-LIMIT FALLBACK
  If Gemini 2.5 Flash hits a 429 mid-stream and no content has been yielded
  yet, textLLM.ts automatically falls back to Claude Sonnet 4.6 so the user
  gets a response rather than an error.

  anthropic         → Claude Sonnet 4.6 for chat (quality/debugging)

  switch — quality matters for long-term memory continuity and vision
