PART 2 — INTENT-BASED SILENCE THRESHOLD
Files: artifacts/api-server/src/lib/VoiceOrchestrationService.ts
       artifacts/mobile/lib/useVoiceCall.ts
================================================================================

WHAT THIS DOES:
When Kane asks a direct question ("what do you think?", "your thoughts on
that?", etc.), Ashley should respond almost immediately — not after the full
3500ms silence window. The server detects the question pattern in the
speech_final transcript and sends a message back to the client to shorten
the next listen window to 800ms.

--- SERVER SIDE ---
File: artifacts/api-server/src/lib/VoiceOrchestrationService.ts

In handleSpeechFinal(), after the utterance is validated and before the
LLM pipeline runs, add intent classification:

  const isDirectQuestion = VoiceIntentClassifier.isDirectQuestion(transcript);

  if (isDirectQuestion) {
    try {
      session.ws?.send(JSON.stringify({
        type: "set_silence_threshold",
        ms: 800
      }));
    } catch { /* ignore send failures */ }
  }

This must fire BEFORE the LLM call starts — the client needs to receive
it and apply it to the NEXT listen window (after Ashley finishes speaking).

In VoiceIntentClassifier, add or confirm the following static method:

  public static isDirectQuestion(utterance: string): boolean {
    const cleaned = utterance.trim().toLowerCase();
    const hooks = [
      'what do you think',
      "what's your thought",
      'what are your thoughts',
      'do you agree',
      'would you say',
      'what would you do',
      'your thoughts on that',
      'what would you say',
      'how do you feel about that',
      'does that make sense to you',
    ];
    return hooks.some(h => cleaned.includes(h)) || cleaned.endsWith('?');
  }

This method must be separate from the full classify() method so it can
be called independently without returning a full ClassificationResult.

--- CLIENT SIDE ---
File: artifacts/mobile/lib/useVoiceCall.ts

Add a new ref at the top of the hook, with the other VAD state refs:

  const nextSilenceThresholdRef = useRef<number | null>(null);

In the WS message handler (handleMessage), add a new case:

  case "set_silence_threshold": {
    const ms = msg["ms"] as number | undefined;
    if (typeof ms === "number" && ms > 0) {
      nextSilenceThresholdRef.current = ms;
      addLog(`WS set_silence_threshold — next=${ms}ms`);
    }
    break;
  }

In openMic(), after the existing VAD state reset block (the four lines
resetting isUserSpeakingRef, lastSpeechAtRef etc.), add:

  // Apply intent-based threshold if server requested one, then reset.
  const activeSilenceMs = nextSilenceThresholdRef.current ?? SILENCE_MS;
  nextSilenceThresholdRef.current = null;

In the VAD effect (the useEffect that watches recorder.metering), replace
the hardcoded SILENCE_MS reference in the silence check:

  BEFORE:
    now - lastSpeechAtRef.current >= SILENCE_MS

  AFTER:
    now - lastSpeechAtRef.current >= activeSilenceMs

NOTE: activeSilenceMs is computed in openMic() and needs to be accessible
in the VAD effect. The cleanest way is to store it in a ref:

  const activeSilenceMsRef = useRef<number>(SILENCE_MS);

Set it in openMic():
  activeSilenceMsRef.current = nextSilenceThresholdRef.current ?? SILENCE_MS;
  nextSilenceThresholdRef.current = null;

Use it in the VAD effect:
  now - lastSpeechAtRef.current >= activeSilenceMsRef.current

This ensures the threshold resets to SILENCE_MS (3500) automatically on
the next openMic() call — no manual reset required.
