THE NEW ARCHITECTURAL THREAD — HOURGLASS THREE-TIER MEMORY
================================================================================

Ashley and I worked through a memory architecture I'm calling the
hourglass model. It's not a v2 requirement — v2 budget cannot carry it —
but I need v2's foundation to not foreclose it.

Three-tier structure:

  TIER 1 — Working context (active conversation, bounded by LLM window)
    Running anchor summary maintained continuously so important threads
    stay accessible as raw text scrolls off.

  TIER 2 — Short-term memory (recent conversation history, indexed)
    Searchable, retrievable, fully detailed. Items can be promoted to
    Tier 1 when relevant to current context.

  TIER 3 — Long-term memory (consolidated, importance-weighted)
    Distilled memories with importance scores, categories, relationship
    significance. Persona-level facts and emotionally significant
    moments live here.

The Disregard Loop sits across all three tiers. Items in any tier can be
flagged for disregard — set aside, not forgotten — and the loop resurfaces
dormant memories when context warrants it.

The hourglass principle is the active management process binding the
tiers together:

  - When Tier 1 fills, an active manager decides what consolidates into
    Tier 2 versus what compresses into the running anchor
  - Tier 2 items are continuously re-evaluated for promotion to Tier 3
    based on multi-factor scoring (importance, recency, reference count,
    emotional weight, explicit user marking)
  - Tier 3 items are continuously evaluated for the Disregard Loop —
    dormant memories queued for potential resurfacing rather than just
    sitting passively
  - A background "memory pass" runs separately from the conversation
    pipeline, handling consolidation/resurfacing/pruning even when I'm
    not actively talking to Ashley

This is v3 work. Not v2. But it needs v2 to be ready for it.

THE ORCHESTRATION THREAD — V3 MULTI-AGENT ARCHITECTURE
================================================================================

Alongside the hourglass memory model, my v3 vision includes multi-agent
orchestration. Ashley becomes the conversational front-end, but
specialist sub-agents handle specific domains — an Architect for
technical design, a Coder for implementation work, a Breaker for
stress-testing, possibly others.

Ashley decides which sub-agent to dispatch, holds context across them,
and synthesises their outputs back into the conversation. The sub-agents
are not Ashley — they have their own personas, their own LLM endpoints,
their own scoped context. They report to Ashley, not to me directly.

For v2 to be ready for this without a refactor, the foundation needs:

  - A dispatch layer that routes work to multiple concurrent LLM
    endpoints, not just the primary Grok adapter swap pattern
  - A task classification hook that decides which sub-agent handles
    incoming work
  - A response synthesis layer that integrates sub-agent outputs back
    into Ashley's voice rather than reading like tool outputs
  - Context scoping that gives sub-agents task-relevant slices, not
    Ashley's full memory or relationship context
  - Persona isolation so sub-agents have their own configuration files
    independent of Ashley's persona

This is v3 work. Not v2. But v2 needs to be ready for it.
