================================================================================
PART 1 — NEW PROTOCOL: EXPLICIT RESPONSE LIFECYCLE
================================================================================

CURRENT (BROKEN) MODEL:
  Server sends sentence chunks → client infers completion from queue state
  → ambiguous between "between sentences" and "response over"

NEW MODEL — four explicit states, server-authoritative:

  1. RESPONSE_START   — server begins generating a response
  2. SENTENCE_END      — server has sent one complete sentence's audio
                          (fires once per sentence, client may start
                          playing immediately — no change to streaming)
  3. RESPONSE_END      — server has sent ALL audio for this response.
                          No more chunks will be sent for this turn.
                          THIS IS THE NEW SIGNAL THAT DISTINGUISHES
                          "between sentences" from "fully done".
  4. PLAYBACK_CONFIRMED — client has finished PLAYING all audio it
                           received. Sent by client only after it has
                           received RESPONSE_END AND its play queue is
                           fully drained AND the player is not active.

THE KEY FIX: RESPONSE_END is sent ONCE, after the LLM stream itself ends
and the final sentence has been flushed to TTS — not after each sentence's
queue drain. The client now has an unambiguous signal for "no more audio
is coming" that is structurally distinct from "the queue is briefly empty
between sentences".

The client must NOT send PLAYBACK_COMPLETE (renamed to PLAYBACK_CONFIRMED
for clarity — see below) until it has received RESPONSE_END. Before
RESPONSE_END, an empty queue means "waiting for more sentences", not
"response complete" — full stop, no exceptions, no timing heuristics.

CHANGE 3: Send RESPONSE_END once, after the LLM stream completes AND
the final sentence has been fully flushed to TTS — replacing the
current premature playback_complete request logic.

  This is the single most important change. Locate the point where the
  LLM stream loop ends (the `for await` over the LLM response, or
  equivalent) and the final sentence buffer has been flushed. After
  that point, and only after that point:

    session.responseComplete = true;
    session.ws?.send(JSON.stringify({
      type: "response_end",
      speechId,
      responseText: session.currentResponseText,
      sequenceNumber: registry.incrementSequence(session),
      timestamp: Date.now(),
    }));

  Do NOT send this from within the per-sentence flush path. It must
  only fire once per turn, after the LLM has finished generating AND
  all TTS chunks have been dispatched to the client.

CHANGE 4: Replace the playback_complete handler logic.

  Rename the inbound message type the server listens for from
  "playback_complete" to "playback_confirmed" (matches new terminology,
  avoids any ambiguity with the old broken signal name).

  handlePlaybackConfirmed() should now simply:
    - Verify session.responseComplete === true. If false, log a
      warning ("playback_confirmed received before response_end —
      ignoring, client should not be able to reach this state") and
      ignore the message. Do not send tts_done in this case.
    - If true: proceed exactly as the old handlePlaybackComplete()
      did — clear the safety timeout, send tts_done, set state to
      listening, process any queued utterance.

CHANGE 5: Fix the safety timeout reference point.
  The 15-second safety timeout (from the original TTS Race Fix) should
  now start when RESPONSE_END is sent, not when the last chunk is sent.
  Move the setTimeout call to CHANGE 3's block, immediately after
  sending response_end.

CHANGE 6: Fix speakFallback() to use the same gate (resolves Issue 2
Cause A from the diagnosis).

FILE: artifacts/api-server/src/routes/voice-call.ts

  speakFallback() currently sends tts_done immediately with no
  confirmation gate. This must be brought into the same lifecycle:

  Replace the immediate tts_done send (lines ~165-179) with:
    1. Send response_end (not tts_done) once the fallback audio
       (e.g. "Still there?") has been fully dispatched.
    2. Set session.responseComplete = true.
    3. Set session.awaitingPlaybackConfirm = true.
    4. Apply the same 15-second safety timeout pattern as CHANGE 5.
    5. Do NOT send tts_done directly. Let the client's eventual
       playback_confirmed message trigger tts_done via the normal
       handlePlaybackConfirmed() path.

  This ensures EVERY path that plays audio to the client — LLM
  responses AND fallback/check-in messages — goes through the same
  lifecycle and the same gate. No more bypass paths.

CHANGE 7: Fix the silence monitor coupling (resolves Issue 2 Cause B).

FILE: artifacts/api-server/src/index.ts

  lastAudioReceivedAt currently only updates on binary mic frames.
  This causes the silence clock to run during the period the client
  is playing audio and the mic is legitimately closed.

  Add a guard to the silence monitor (wherever silenceMs is calculated
  and compared against the warning/check-in threshold):

    Do not evaluate or act on the silence threshold if
    session.state !== "listening" OR session.awaitingPlaybackConfirm
    is true.

  The silence clock should only be meaningfully measuring "Kane has
  gone quiet while we are actively listening for him" — not measuring
  time while Ashley is speaking or while waiting for playback
  confirmation. This is a conceptual fix: silence detection must be
  scoped to genuine listening windows only.

WHAT NOT TO DO
================================================================================

- Do not reintroduce any timing-based heuristic ("wait X ms then assume
  done"). The whole point of this brief is to replace timing guesses
  with explicit signals.
- Do not leave any audio-sending code path (LLM response OR fallback
  messages) outside the new lifecycle. Every path must go through
  RESPONSE_START → SENTENCE_END* → RESPONSE_END → PLAYBACK_CONFIRMED.
- Do not change VAD constants (SILENCE_DB, SILENCE_MS, MIN_SPEECH_MS) —
  not in scope for this brief.
- Do not touch the intent classification or context assembly logic —
  not in scope for this brief.
