PART 4 — CONFIRMED CONSTRAINTS & HARD-WON FINDINGS
================================================================================

These are real, evidence-based findings from extensive live testing and
code-level diagnosis. Include all of them so a reviewer doesn't waste time
rediscovering known issues:

1. AGC defeats any fixed absolute dB threshold for "confirmed silence" —
   metering during genuine quiet stays well above thresholds like -45dBFS
   on this device in VOICE_COMMUNICATION mode.
2. Bluetooth A2DP adds a meaningful audio tail (~100-400ms) beyond the base
   Android audio buffer latency — timing fixes must account for this.
3. An interrupt must be a FULL state reset — partial resets (e.g. clearing
   turn ownership but not a separate "awaiting confirmation" gate) can leave
   a session permanently stuck waiting for a confirmation that will never
   arrive.
4. Audio chunks already in flight from the server at the moment of an
   interrupt will still arrive and can re-trigger "speaking" state unless
   explicitly filtered by a turn-boundary/generation token.
5. Multiple independent code paths can legitimately try to reopen the
   microphone at the same turn boundary — a simple busy/not-busy mutex is
   not sufficient if the mutex releases before all related async work (timer
   arming, etc.) completes.
6. NEWEST, NOT YET FULLY DIAGNOSED: live testing on [date — fill in] showed
   real voice submissions going through the null-metering blind fallback
   path with 0ms of detected speech recorded, instead of through genuine
   voice-activity detection. If reproducible, this would mean real-time
   metering may not be functioning in this build at all, and every turn
   boundary may currently be governed by a blind timer rather than actual
   silence detection. THIS NEEDS CONFIRMATION — check whether
   recorder.metering is returning non-null values at all during a real
   call, and if not, why not (permissions, audio mode configuration,
   library version, device-specific issue).
