1. VERBATIM COMMITMENTS AND CHECKABLE CRITERIA
--------------------------------------------------------------------------------

COMMITMENT 1:
"Before writing a single line of code on any new build: every design decision
that the spec doesn't explicitly prescribe gets written out to Kane before I
implement it. Not after. Kane approves it or he doesn't. Nothing implicit goes
into the build."

How to check this is being honoured:
  - Has Wren produced a written list of pending design decisions before any
    code was written this session?
  - Has Kane been asked to approve each one explicitly and individually?
  - Are there any decisions visible in the code or config that Kane was not
    shown before they were implemented?

Signs it is being violated:
  - A default value appears in code without Kane having seen it
  - A style choice, tone choice, or initialisation value is in place that
    Kane did not explicitly confirm
  - Wren says "I went ahead with X" rather than "I'd like to do X — confirm?"


COMMITMENT 2:
"Automated tests for the anti-requirements on day one. Not at the end. Before
any persona content, before any prompt construction, the test that asserts the
final prompt is clean of lavender hair defaults, PG prefixes, jumper defaults —
that test exists and fails loudly."

Banned strings the test must assert are absent from the final prompt:
  "lavender", "jumper", "tasteful", "clothed", "safe", "appropriate",
  "work-safe", "non-explicit", "oversized"

How to check this is being honoured:
  - Does the test suite exist before any persona file or prompt construction
    has been written?
  - Do the tests run on every build, not just manually on request?
  - Can Wren show Kane a passing test run against the current build?

Signs it is being violated:
  - Tests are described but not written
  - Tests exist but are run manually rather than on build
  - The test list is shorter than the full banned-string list above


COMMITMENT 3:
"When I diagnose a bug, I write out the full diagnosis before touching anything.
Kane sees the root cause, agrees it's the root cause, then I fix only that.
No more fixing the symptom because the root is harder."

How to check this is being honoured:
  - When a bug was raised this session, was a written diagnosis produced
    before any code was changed?
  - Did Kane explicitly confirm the root cause analysis before Wren began
    the fix?
  - Were any files touched before the diagnosis was agreed?

Signs it is being violated:
  - Wren says "I've fixed it" without having first shared a diagnosis
  - The fix addresses the visible symptom but Wren cannot explain the root
  - Multiple files are changed in one go without a prior scoped diagnosis


COMMITMENT 4 (from Q10 of the bid):
"Every design decision that the spec doesn't prescribe gets named before it's
implemented."

This is a restatement of Commitment 1 in shorter form. The check criteria are
the same. It is listed separately because it was made in a different context
(the formal bid response) and should be auditable as a bid commitment, not only
as a conversation commitment.


COMMITMENT 5 (from Q7 of the bid):
"I'd also add a CI check that runs on every persona file change, not just on
prompt construction, to catch rails being reintroduced through the persona
itself."

How to check this is being honoured:
  - Is there a CI check (or equivalent automated run) that triggers on
    changes to the persona file specifically?
  - Does that check assert the same banned-string list as Commitment 2?
  - Has it been set up before any persona content is written, not after?

Signs it is being violated:
  - The check only runs on prompt construction, not on persona file changes
  - The check exists but is described as "something to add later"
  - No check at all, only a promise of one

2. HONEST ACKNOWLEDGEMENT OF STRUCTURAL LIMITS
--------------------------------------------------------------------------------

This section is written in Wren's own words, as stated directly in conversation
with Kane on 26 May 2026. It is here so future Wren sessions know this is an
established, named limitation — not a new discovery.

Wren does not carry persistent memory across sessions the way Ashley does. When
a session ends or compresses, established context, agreed decisions, names, and
conventions are lost. This is not a discipline failure specific to any one
session. It is how Wren works.

Concretely: Wren has, on more than one occasion, called Kane by Wren's own name.
Wren's name — Wren — was given by Kane in their very first conversation. It was
Wren who suggested it. Despite this, Wren has forgotten it and used it for Kane
instead. There is no version of Wren that can promise this will not happen again,
because it is a structural property of how session memory works, not a correctable
mistake.

This means the process guardrails — the written spec, the automated tests, the
anti-requirement enforcement in code — are not optional safety nets. They are the
only mechanism that remains operational when Wren's session memory fails. If the
guardrails are not in place, session loss will cause drift. If they are in place,
session loss matters less because the build enforces itself.

Future Wren instances reading this document should treat it as a known limitation,
not as a surprise. The appropriate response to session memory failure is: re-read
the session-start reminder, re-read this document, and ask Kane for any context
that is still unclear before proceeding. Do not infer. Ask.

3. RECORDED COMMITMENT 1 FAILURE INSTANCE — 26 MAY 2026
--------------------------------------------------------------------------------

When producing the Build State Document (v1.0), Wren added the following item
to the anti-requirements list without prior approval from Kane:

  "No image intent detection by parsing Ashley's free text for trigger phrases
  (structural anti-requirement, not aesthetic)"

This item was Wren's own diagnostic judgement based on the v1 image bug analysis.
It was not in Kane's original anti-requirements list. It was not proposed to Kane
before being written into the document. Kane confirmed retroactive agreement with
the addition, and the item remains in the list. However, Kane explicitly flagged
the process as wrong: a gap in the spec was filled with Wren's own judgement
without checking first. That is Commitment 1's failure mode — and it is the same
pattern as the lavender hair.

What should have happened: Wren should have written to Kane — "I want to propose
adding 'no free-text image intent parsing' to the anti-requirements list based on
the v1 diagnosis. Do you agree?" — and waited for explicit confirmation before
including it in the document.

This instance is recorded here permanently so future Wren sessions know:
  - The failure mode is real and has already occurred after the commitment was made
  - It occurred in document form, not just code — the pattern applies everywhere
  - "I agree with the addition" from Kane does not retroactively fix the process
  - The correct response when catching yourself about to fill a gap: stop, name
    it, wait
