ASHLEY PROJECT — FIX BRIEF FOR WREN
Task: Separate auxiliary clip lifecycle from main conversation turn lifecycle
Version: 1
Date: 19 June 2026
================================================================================

ROOT CAUSE (confirmed by your diagnosis + independent review):

The "thinking" filler clip and the main LLM response share the same
client-side lifecycle signals (speech_start, tts_done). The client cannot
tell "auxiliary clip finished" apart from "main response finished". When
the thinking clip's tts_done arrives, the client wrongly closes out the
entire turn and opens the mic before Ashley's real reply has even started
generating.

THIS IS NOT A TIMING BUG. IT IS A MISSING DISTINCTION. The fix gives
auxiliary audio (thinking clip, and any future filler clips) its own
identity that the client recognises and ignores for turn-completion
purposes, while still playing it audibly and promptly.

================================================================================
REQUIREMENTS (carried over from your diagnosis — all six apply)
================================================================================

REQ-1: Client lifecycle gates (ttsServerDoneRef, responseEndReceivedRef,
        phase transitions) must not be affected by non-main-turn audio.
REQ-2: Thinking clip must still play audibly and promptly.
REQ-3: Server decides what is "main" vs "auxiliary" — client must not
        hard-code speechId knowledge.
REQ-4: 15-second safety timeout remains a backstop for genuine lost-
        connection cases only — not the normal path.
REQ-5: Must work whether the thinking clip fires or not. No client-side
        special-casing on "did the watchdog fire".
REQ-6: The existing response_end → playback_confirmed → tts_done lifecycle
        for the MAIN turn is preserved without modification.

================================================================================
THE FIX — explicit audio kind tagging
================================================================================

Every message that carries or announces audio gets a new field: `kind`.

  kind: "main"        — the actual conversation response (LLM-generated)
  kind: "auxiliary"    — filler audio: thinking clip, future check-ins,
                          any clip that is NOT the conversational answer

The client uses `kind` to decide whether a given speech_start / tts_done
pair is allowed to touch the main lifecycle refs. Auxiliary audio plays
through its own lightweight, isolated path that never touches
ttsServerDoneRef, responseEndReceivedRef, or phase.

PART 1 — SERVER CHANGES
================================================================================

FILE: artifacts/api-server/src/lib/VoiceOrchestrationService.ts

CHANGE 1: Add `kind` to every speech_start, sentence_end, tts_done, and
response_end message.

  Main turn messages (speech_start at line ~500-514, response_end at
  line ~638, and the per-sentence sentence_end sends): add `kind: "main"`.

  speakClipOrText() messages (lines 131-220, used for the thinking clip
  and any other pre-loaded clip): add `kind: "auxiliary"`.

CHANGE 2: Stop speakClipOrText() from touching session.currentSpeechId
at all.

  Currently speakClipOrText() saves currentSpeechId, temporarily
  overwrites it with a clip-specific ID, then restores it. This was the
  mechanism that allowed the clip to interfere with gate checks elsewhere
  in the codebase.

  Remove this save/overwrite/restore pattern entirely. speakClipOrText()
  should use its own locally-scoped speechId for the clip's messages
  (e.g. a fresh UUID generated at call time) and must NEVER assign to
  session.currentSpeechId. The main turn's currentSpeechId stays
  untouched throughout — there is no longer any reason for the clip to
  borrow or interact with it, since the client will now distinguish by
  `kind`, not by speechId matching.

CHANGE 3: Confirm response_end (line ~638) continues to use
session.currentSpeechId and kind: "main" exactly as before. No other
change needed here — this path was already correct per your diagnosis.

PART 2 — CLIENT CHANGES
================================================================================

FILE: artifacts/mobile/lib/useVoiceCall.ts

CHANGE 4: speech_start handler (lines ~483-509) — only reset main
lifecycle refs when kind === "main".

  case "speech_start": {
    const kind = msg["kind"] as string | undefined;
    if (kind === "main" || kind === undefined) {
      // undefined fallback: treat as main for backward compatibility
      // during rollout, remove fallback once server is confirmed always
      // sending kind.
      ttsServerDoneRef.current = false;
      responseEndReceivedRef.current = false;
      phaseRef.current = "speaking";
      // ... existing main-path logic unchanged
    } else {
      // auxiliary clip — do not touch any main lifecycle ref.
      addLog(`WS speech_start (auxiliary, kind=${kind}) — playing without affecting main lifecycle`);
    }
    break;
  }

CHANGE 5: tts_done handler (lines ~511-521) — only set
ttsServerDoneRef when kind === "main".

  case "tts_done": {
    const kind = msg["kind"] as string | undefined;
    if (kind === "main" || kind === undefined) {
      ttsServerDoneRef.current = true;
      void playNextRef.current();
    } else {
      addLog(`WS tts_done (auxiliary, kind=${kind}) — ignored for main lifecycle`);
      // Auxiliary clip audio itself must still have already been queued
      // and played via its own playback path — see CHANGE 7.
    }
    break;
  }

CHANGE 6: sentence_end and response_end handlers — same kind check
pattern. Only act on main-lifecycle refs (responseEndReceivedRef, etc.)
when kind === "main". Auxiliary sentence_end/response_end (if ever sent
for a clip) should be logged and otherwise ignored for lifecycle purposes.

CHANGE 7: Auxiliary clip audio playback path.

  Auxiliary clip binary audio must still be queued and played — REQ-2
  requires it plays audibly and promptly. Use a SEPARATE, simple playback
  mechanism for auxiliary clips that does not route through the main
  playQueueRef / playNext() state machine at all.

  Recommended: a lightweight dedicated player instance for auxiliary
  clips (e.g. auxPlayerRef), fired immediately on receipt of auxiliary
  binary audio, fire-and-forget, no queue, no draining logic, no
  interaction with phase or any *Ref used by the main lifecycle.

  This guarantees REQ-1 structurally — auxiliary audio physically cannot
  touch main lifecycle state because it does not go through the same
  code path at all.
