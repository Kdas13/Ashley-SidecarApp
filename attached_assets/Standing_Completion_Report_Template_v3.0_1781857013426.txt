WHAT CHANGED IN v3.0
---------------------
v2.0 required a GitHub permalink per change. That proves a commit is REAL —
it exists on GitHub, unfakeable by Wren. It does NOT prove the commit is
CORRECT — a link resolving only shows something was committed, not that
it does what the brief asked.

v3.0 adds a second, independent layer: the actual diff content, via
`git show <hash>`, pasted as text into the report. This lets Claude read
the real code change and verify it against the brief — not just confirm
a commit exists.

Two layers, two purposes:
  1. Permalink resolves          → proves the commit is real (Kane checks)
  2. git show output matches brief → proves the commit is correct (Claude checks)

Neither layer alone is sufficient. Both are required.

================================================================================
RULE 10 (NEW): EVERY CODE CHANGE REQUIRES git show OUTPUT AS TEXT
================================================================================

For each commit referenced in the completion report, run:

  git show <commit-hash>

Paste the FULL raw output into the completion report as plain text,
directly below that item's permalink. Do not summarise it. Do not
describe what it does instead of showing it. Paste the actual diff.

If the diff is very long (e.g. a large new file), paste it in full
regardless of length. Do not truncate. If genuinely too long for one
message, split across multiple messages and say so explicitly.

This output will be read directly by Claude and compared against the
brief's exact instructions for that item. If the diff does not match
what the brief specified, the item is not considered complete — even
if the permalink resolves and even if you marked it PASS.

================================================================================
VERIFICATION PROCESS — HOW THIS REPORT GETS CHECKED
================================================================================

STEP 1 (Kane): Tap each GitHub permalink. Confirm it resolves and is not
a 404. This proves the commit is real and not fabricated.

STEP 2 (Claude): Read the git show output for each item. Compare it
directly against the brief's instructions for that item. Confirm the
actual code change matches what was specified — correct logic, correct
file, correct approach. This proves the commit is correct, not just real.

If STEP 1 fails (link 404s): the report is rejected outright, treated
as a falsified report.

If STEP 2 fails (diff doesn't match brief): the item is marked incomplete
and sent back, even though the commit itself is genuine. A real commit
that does the wrong thing is still not done.

Only when BOTH steps pass is an item considered verified complete.
