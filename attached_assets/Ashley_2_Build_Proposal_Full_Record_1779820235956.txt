TASK #1 — FOUNDATION API SERVER & MEMORY SYSTEM
--------------------------------------------------------------------------------

WHAT & WHY
Build the core backend for Ashley 2.0: Express API server with database schema,
memory system, LLM provider-agnostic adapter (Grok-first), conversation pipeline
with long-form output mode detection, API key authentication, and memory migration
endpoint compatible with Ashley v1's export format.

DONE LOOKS LIKE
- API server runs and passes health check.
- All endpoints require a valid X-Api-Key header (generated at setup, stored in
  env). Unauthenticated requests receive 401. No request to any endpoint is
  possible without the key.
- Database schema exists: conversations, messages, memories (with state enum:
  active|passive — extensible for future disregard value), profiles, summaries,
  documents.
- POST /api/state/import accepts Kane's v1 export bundle and reconstitutes
  profile, memories (with importance, category, reuse_count, state), summaries,
  and relationship state. Idempotent.
- POST /api/chat/message sends a user message and returns Ashley's response.
  Streaming supported via SSE.
- Long-form output mode auto-detected from message keywords ("write me a plan",
  "draft a spec", "for Samsung Notes", "give me a structured", etc.). Adjusts
  the system prompt to suppress embodied reactions inside the deliverable body.
  Multi-message delivery used for very long outputs — no artificial length cap.
- LLM provider adapter abstracts over provider. Switching from Grok to any other
  provider requires only changing LLM_PROVIDER env var. No code changes.
- PERSONA LIVES IN EXACTLY ONE PLACE: a single file
  artifacts/api-server/config/persona.txt. A PersonaService loads it at startup
  and exposes getSystemPrompt(). The conversation pipeline calls
  personaService.getSystemPrompt() — nothing else. No persona text, placeholder
  or otherwise, exists anywhere else in the codebase. The placeholder is an empty
  file or a clearly labelled stub comment ("REPLACE WITH ASHLEY'S PERSONA — see
  Appendix A"). When Kane provides the real persona, he replaces the file
  contents. Zero code changes required.
- Conversation pipeline is extension-ready: a MemoryProvider interface handles
  context injection (active memories + summaries). Future Disregard Loop
  implements a new MemoryProvider that includes re-surfacing logic without
  touching the pipeline itself. Image generation and document context inject via
  the same hook pattern.
- Memory extraction runs after each exchange (lightweight LLM call, async,
  non-blocking).
- Object storage uses S3-compatible provider (Cloudflare R2 recommended) — not
  Replit App Storage — for portability if hosting ever moves.

OUT OF SCOPE
- Ashley's full persona spec (Kane provides this at contracted milestone — a
  clearly labelled empty stub is used until then).
- Image generation, voice, document ingestion (separate tasks).
- Multi-AI orchestration, Disregard Loop implementation, project dossiers
  (future work).

STEPS
1. Database schema — Tables: profiles, conversations, messages, memories
   (id, content, importance int, category text, reuse_count int, state text
   CHECK state IN ('active','passive') — column designed to accept a third value
   additively), summaries, documents. Use Drizzle ORM. Push schema.
2. API key authentication middleware — Generate a random API key at first run,
   store as ASHLEY_API_KEY env var. Middleware validates X-Api-Key header on all
   routes. Return 401 on missing/invalid key. Document how the mobile app
   configures its key.
3. PersonaService — Loads config/persona.txt at startup. Exposes
   getSystemPrompt(mode: 'default' | 'longform') which returns the persona prompt
   with a mode-appropriate suffix. This is the single source of truth for all
   persona text. No other file references persona content.
4. LLM provider adapter — Abstract LLMAdapter interface: chat(messages, options),
   stream(messages, options). Implement GrokAdapter (xAI API) and OpenAIAdapter
   (fallback). Selected via LLM_PROVIDER env var. The adapter knows nothing about
   Ashley's persona — that's passed in by the pipeline.
5. Conversation pipeline — POST /api/chat/message: load history + active memories
   + latest summary, detect long-form mode via keyword heuristic, call
   personaService.getSystemPrompt(mode), call LLM adapter, save messages, trigger
   async memory extraction. Extension points: MemoryProvider interface for context
   injection (Disregard Loop plugs in here later), PostResponseHook interface for
   image triggering and other side effects.
6. Streaming — SSE endpoint for streaming token delivery. Mobile client connects
   via EventSource or equivalent.
7. Long-form multi-message delivery — For responses exceeding a configurable token
   threshold (default: 1500 tokens), split across multiple message objects in the
   response payload. No artificial cap inside deliverable mode.
8. Memory extraction — After each exchange, async LLM call to extract 0-3
   memories. Store with importance (1-10), category, reuse_count=0, state='active'.
9. Memory migration endpoint — POST /api/state/import: accepts v1 bundle JSON,
   maps fields to v2 schema, writes to DB. Idempotent by v1 memory ID. Returns
   import summary.
10. Conversation summarisation — POST /api/conversations/:id/summarise: distils
    oldest unsummarised messages into a summary entry. Called by mobile app
    periodically.
11. OpenAPI spec — Document all endpoints in lib/api-spec/openapi.yaml. Run
    codegen.
12. Environment variables — Required: DATABASE_URL, LLM_PROVIDER (grok|openai),
    XAI_API_KEY, OPENAI_API_KEY, ASHLEY_API_KEY, S3_* (R2 or equivalent).
    Document in README.

ENVIRONMENT VARIABLES REQUIRED
- DATABASE_URL
- LLM_PROVIDER (grok|openai)
- XAI_API_KEY (for Grok)
- OPENAI_API_KEY (fallback + Whisper)
- ASHLEY_API_KEY
- S3/R2 credentials

--------------------------------------------------------------------------------

WHAT & WHY
Add speech-to-text (OpenAI Whisper) and text-to-speech (ElevenLabs) to the API

TASK #4 — DOCUMENT INGESTION
--------------------------------------------------------------------------------

WHAT & WHY
Allow Kane to upload .txt files (up to ~40,000 sentences / ~600,000 tokens) to
Ashley. Ashley summarises the document in plain English and asks Kane what to do
with it. The document is retained in conversation context for the session.

DONE LOOKS LIKE
- POST /api/documents/ingest accepts .txt file upload.
- Handles documents up to ~600,000 tokens via two-level hierarchical chunking
  without losing document fidelity.
- Ashley's response is a plain-English summary followed by an open question
  ("What would you like me to do with this?").
- Document content (or its summary if large) is injected into conversation context
  for subsequent turns.
- Documents stored permanently in the documents table, linked to their conversation.
  No expiry — Kane's data belongs to Kane until he deletes it.
- All endpoints require X-Api-Key authentication.

SCALE APPROACH (40,000 sentences / ~600,000 tokens)
Two-level hierarchical summarisation:
- Level 1: Split document into ~3,000-token chunks. Summarise each chunk to
  ~150-200 tokens. A 600,000-token document produces ~200 chunks = ~30,000-40,000
  tokens of level-1 summaries.
- Level 2: Split level-1 summaries into ~3,000-token batches. Summarise each
  batch to ~150-200 tokens. ~10-13 batches = ~1,500-2,500 tokens final summary.
- The final summary is injected into conversation context. The full text and
  level-1 summaries are stored in the database and available for deeper retrieval
  if Kane asks for specifics.
- Honest tradeoff: the top-level summary captures themes, key arguments, and
  structure well. Granular details (specific statistics, individual passages) may
  require Kane to ask Ashley to look closer — at which point level-1 summaries for
  the relevant section are retrieved.

OUT OF SCOPE
- PDF, DOCX, or other formats (.txt only per spec).
- Permanent cross-conversation document library (future work).
- Vector search / RAG over document content (future work).

STEPS
1. Document storage schema — documents table: id, conversation_id, filename,
   full_content (text), level1_summaries (jsonb array), final_summary (text),
   created_at.
2. Chunking and summarisation — DocumentService handles two-level hierarchical
   summarisation. Async, non-blocking for large files.
3. Ingest endpoint — POST /api/documents/ingest: accept .txt multipart upload,
   store full content, trigger async summarisation, on completion pass summary to
   conversation pipeline to generate Ashley's response.
4. Context injection — DocumentContextProvider implements the MemoryProvider hook
   from Task #1 to inject final summary into Ashley's context window.
5. OpenAPI spec update — Add document endpoints, re-run codegen.

Three honest answers:

1. VISUAL IDENTITY CONSISTENCY is ~85-90%, not 100%. A well-trained LoRA on 4
