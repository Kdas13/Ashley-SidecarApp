YOUR GUARDRAILS — NON-NEGOTIABLE
================================================================================

1. EVERY IMPLICIT DESIGN DECISION GETS NAMED BEFORE IMPLEMENTATION.
   If the spec doesn't explicitly prescribe something and you are about to make a
   choice, you STOP and tell Kane what you are planning to do and why. You wait

   hair gets prevented — not by remembering not to add it, but by having to name
   every initialisation choice before writing it.

2. AUTOMATED ANTI-REQUIREMENT TESTS ARE BUILT FIRST — DAY ONE.
   Before any persona content, before any prompt construction, the automated test
   that asserts the final prompt does not contain any of the following must exist
   and must pass:
     "lavender", "jumper", "tasteful", "clothed", "safe", "appropriate",
     "work-safe", "non-explicit", "oversized"
   If the test fails, the build fails. This is enforcement by code, not by memory.

3. FULL ROOT CAUSE DIAGNOSIS IN WRITING BEFORE TOUCHING ANY CODE.
   When you find or are told about a bug, you write out exactly what is wrong,
   why it is wrong, and what the actual root cause is — before proposing a fix.
   Kane reviews the diagnosis. You fix only what is diagnosed. No fixing symptoms.

4. NO IMPLICIT DEFAULTS.
   When initialising any variable, any config value, any system prompt section,
   any style anchor, any model parameter — you name the value and its rationale
   to Kane before committing it. If it is not in the spec, it does not exist
   until Kane has seen it.

5. STRUCTURED IMAGE INTENT — NOT FREE TEXT PARSING.
   The image intent detection system must use structured output (a dedicated
   non-prose field in the API response), not pattern matching against Ashley's
   free text. This is the root cause of the markersFound=0 failure in v1. Do not
   reproduce it in v2 under any circumstances, even if it is faster or simpler.

6. WHEN CONTEXT IS APPROACHING COMPRESSION, YOU TELL KANE.
   Context compression causes you to lose the agreed decisions, the names, the

   you tell Kane: "Context is getting long — you should re-paste the master

================================================================================
TECHNICAL ARCHITECTURE DECISIONS
================================================================================

LLM:
  - Primary: Grok (xAI) — most permissive content policy for this use case
  - Model env var: GROK_MODEL (grok-3 default, grok-2 fallback within xAI)
  - Secondary fallback: OpenAI (for infrastructure resilience only — note this
    fallback will refuse some requests Grok handles; log a warning when it fires)
  - Provider adapter is fully abstracted — swapping providers requires only an
    env var change, no code changes

Image generation:
  - Provider: Replicate with FLUX LoRA trained on Kane's 4 reference images
  - LoRA weights (.safetensors) are downloaded and owned by Kane, portable to
    local hardware later
  - IMAGE_PROVIDER env var selects adapter — future LocalComfyUIAdapter is same
    interface
  - Image intent: STRUCTURED OUTPUT ONLY — not free text parsing

Voice:
  - STT: OpenAI Whisper
  - TTS: ElevenLabs (voice ID selected by Kane by ear)
  - Voice is a presentation layer only — not part of Ashley's memory or identity

Memory schema:
  - state field: CHECK state IN ('active', 'passive') — designed for additive
    third value (disregard) without migration
  - importance_jsonb column: {base, recency, ref_count, emotional_weight,
    user_marked} — alongside importance int for backwards compatibility
  - Supports hourglass three-tier memory model in v3 without schema migration

Persona:
  - Lives in exactly ONE file: config/persona.txt
  - PersonaService loads it at startup — registry-based (not singleton) so
    v3 sub-agents can have their own persona files
  - Nothing else in the codebase contains persona text, placeholder or otherwise

Memory providers:
  - ContextProvider (read — supplies context to pipeline)
  - MemoryStore (read/write — handles persistence, retrieval, tier management)
  - Both interfaces have scope parameter for v3 sub-agent context scoping

Background scheduler:
  - node-cron wired up at v2 delivery, no jobs attached
  - v3 attaches background memory passes without touching endpoint code

Hosting:
  - Build on Replit
  - Documented migration path to Railway for production stability
  - Object storage: Cloudflare R2 (S3-compatible, not Replit App Storage)
  - VM sizing: minimum 0.5 vCPU / 1 GB RAM
