WHAT THIS IS
------------
A new set of admin REST endpoints added to Ashley's existing Replit Node.js
backend. These endpoints allow the Base44 dashboard (Ashley Command Centre) to
read and control Ashley's backend without touching the existing HMAC device
authentication system.

This is NOT the full Ashley build. This is a thin admin API layer that sits
alongside the existing server. The core logic already exists — Wren is mostly
connecting existing data to new routes with a different auth method.

WHAT ALREADY EXISTS (do not rebuild these)
------------------------------------------
- Node.js Express server on Replit
- Postgres database with messages, memories, feature flags tables
- Worker heartbeat system (workers_heartbeat table)
- GET /health endpoint returning queue depths and worker status
- Image move worker and storage purge worker
- HMAC device authentication middleware (hmacAuth)

WHAT WREN NEEDS TO BUILD
-------------------------
1. Admin API key authentication middleware
2. /admin/* route handlers (listed below)
3. Ashley-to-Base44 write capability (Base44 SDK integration)
4. Store Base44 Service Role Key in Replit Secrets

================================================================================
STEP 1 — ADMIN API KEY MIDDLEWARE
================================================================================

File: src/middleware/adminAuth.ts

All /admin/* routes must check for this header:
  X-Admin-API-Key: <value from ADMIN_API_KEY Replit Secret>

Implementation:

  import { Request, Response, NextFunction } from "express";

  const ADMIN_API_KEY = process.env.ADMIN_API_KEY?.trim();
  if (!ADMIN_API_KEY) {
    console.error("STARTUP FATAL: ADMIN_API_KEY not set"); process.exit(1);
  }

  export function adminAuth(req: Request, res: Response, next: NextFunction): void {
    const key = (req.headers["x-admin-api-key"] as string ?? "").trim();
    if (!key || key !== ADMIN_API_KEY) {
      res.status(401).json({ error: "unauthorized" }); return;
    }
    next();
  }

Add to Replit Secrets:
  ADMIN_API_KEY = <generate a random 32-char string>

Apply adminAuth middleware to ALL /admin/* routes.
This is completely separate from the existing HMAC device auth.
Do NOT modify existing HMAC routes.

--- QUICK COMMANDS (no request body, return { "success": true }) ---

  UPDATE image_move_queue SET status='pending', locked_at=NULL, worker_id=NULL

  Return { "success": true, "cleared": N }

  Return { "success": true }

POST /admin/health/full
  Run all 6 health checks and return combined results.
  See HEALTH CHECK RESPONSE SCHEMA below.

  Clear any in-memory caches (signed URL cache, capabilities cache).
  Return { "success": true }

--- HEALTH CHECKS (all return same schema) ---

HEALTH CHECK RESPONSE SCHEMA:
  {
    "status": "pass" | "warn" | "fail",
    "summary": "one-line string shown in dashboard",
    "findings": [
      {
        "level": "info" | "warning" | "error",
        "message": "string",
        "action": "string (optional suggested fix)"
      }
    ]
  }

POST /admin/health/memory
  Check: query memories table for NULL content, invalid categories,
  zero reuse_count entries older than 30 days.
  Return findings per item.

POST /admin/health/memory-integrity
  Check: look for orphaned memory entries, duplicate memory content
  (Jaccard similarity > 0.9), entries missing required fields.
  Return findings.

POST /admin/health/prompts
  Check: are any Tier 2 prompts empty, any Tier 3 templates with no content.
  Return findings.

POST /admin/health/providers
  Check: ping each configured provider API (Gemini, Claude, ElevenLabs, Deepgram).
  For each: response time in ms, status pass/warn/fail.
  Return findings.

POST /admin/health/workers
  Check: workers_heartbeat table. Flag any worker with stale=true or last_run_ok=false.
  Return findings.

POST /admin/health/context-size
  Estimate: count tokens in system prompt + active memories + recent conversation.
  Return estimated token count and percentage of context window used.
  Use rough estimate (1 token ≈ 4 chars) if no tokeniser available.

--- MEMORY ---

GET /admin/memories?limit=100&offset=0&category=&state=&search=
  SELECT id, content, category, importance_weight, state, reuse_count,
         created_at, updated_at
  FROM memories
  WHERE ($category = '' OR category = $category)
    AND ($state = '' OR state = $state)
    AND ($search = '' OR content ILIKE '%' || $search || '%')
  ORDER BY updated_at DESC
  LIMIT $limit OFFSET $offset
  Return: { memories: [...], total: N }

POST /admin/memory/add
  Body: { content: string, category: string, importance_weight: number }
  INSERT INTO memories (content, category, importance_weight, state, reuse_count)
  VALUES ($content, $category, $importance_weight, 'active', 0)
  Return: { success: true, id: newId }

POST /admin/memory/update
  Body: { id: string, content?: string, category?: string,
          importance_weight?: number, state?: string }
  UPDATE memories SET ... WHERE id = $id
  Return: { success: true }

DELETE /admin/memories/:id
  UPDATE memories SET state = 'deleted', updated_at = NOW() WHERE id = $id
  (soft delete — do not hard delete)
  Return: { success: true }

--- CONVERSATIONS ---

GET /admin/conversations?limit=100&offset=0&search=&from=&to=

  FROM conversations (or equivalent sessions table)
  ORDER BY created_at DESC
  LIMIT $limit OFFSET $offset

GET /admin/conversations/:id

DELETE /admin/conversations/:id

STEP 3 — ASHLEY WRITES TO BASE44
================================================================================

Ashley's backend needs to write inbox messages and proposals directly to Base44's
database using the Base44 SDK.

Install Base44 SDK:
  npm install @base44/sdk

Configure (add to Replit Secrets):
  BASE44_SERVICE_ROLE_KEY = <get this from Base44 project settings panel>

Usage (src/lib/base44Client.ts):

  import { createClient } from "@base44/sdk";

  const base44 = createClient({
    serviceRoleKey: process.env.BASE44_SERVICE_ROLE_KEY!,
  });

  export async function postInboxMessage(
    content: string,
    priority: "info" | "warning" | "urgent",
    referenceCode: string,
  ): Promise<void> {
    await base44.entities.InboxMessage.create({
      content,
      priority,
      reference_code: referenceCode,
      status: "unread",
      created_at: new Date().toISOString(),
    });
    // If priority === "urgent": Base44 automatically fires email notification
  }

  export async function postProposal(
    proposalType: string,
    proposedChange: object,
    reasoning: string,
  ): Promise<void> {
    const VALID_TYPES = [
      "add-memory", "modify-memory", "change-flag",
      "change-provider", "new-tier3-prompt", "adjust-spend-threshold"
    ];
    if (!VALID_TYPES.includes(proposalType)) {
      throw new Error(`Invalid proposal type: ${proposalType}`);
    }
    await base44.entities.Proposal.create({
      proposal_type: proposalType,
      proposed_change: JSON.stringify(proposedChange),
      reasoning,
      status: "pending",
      created_at: new Date().toISOString(),
    });
  }

When to call postInboxMessage:
  - Worker error after 3 consecutive failures -> priority "warning"
  - Spend cap threshold approached (>80%) -> priority "warning"
  - Backend error rate spike -> priority "urgent"
  - Memory consolidation complete -> priority "info"

When to call postProposal:
  - Ashley detects a memory that should be updated -> "modify-memory"
  - Ashley detects a flag that should be toggled -> "change-flag"
  (Only call these when Ashley's orchestration layer is active — Phase 2)

  CREATE TABLE IF NOT EXISTS spend_caps (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    category    TEXT NOT NULL DEFAULT 'general',
    enabled     BOOLEAN NOT NULL DEFAULT false,
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );

  -- Prompts (if not already exists)
  CREATE TABLE IF NOT EXISTS prompts (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tier        INTEGER NOT NULL CHECK (tier IN (1,2,3)),
    name        TEXT NOT NULL,
    content     TEXT NOT NULL,
    description TEXT,
    version     INTEGER NOT NULL DEFAULT 1,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  );
