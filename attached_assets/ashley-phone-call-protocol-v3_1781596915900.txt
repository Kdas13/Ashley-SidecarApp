THE VISION
================================================================================

The goal is a "phone call" mode for Ashley — a hands-free, continuous voice
conversation where the user speaks naturally, Ashley responds in her voice
(Daisy), and the loop continues automatically without the user touching the
phone. Designed specifically for car use (phone in cupholder) but applicable
anywhere hands-free conversation makes sense.

Reference implementations that achieve this experience:
- Replica AI (phone call feature)
- Claude voice (iOS/Android)
- ChatGPT Advanced Voice Mode
- Grok voice mode

All of these use a persistent WebSocket connection with streaming audio in
both directions and server-side voice activity detection (VAD) to handle
natural pauses. This is the same pattern being implemented here.

Key UX requirements:
1. One button to START a call. Zero buttons during the call.
2. Ashley listens automatically after she finishes speaking.
3. User can interrupt Ashley mid-reply (she stops, listens).
4. Works with screen locked/dimmed (car cupholder use case).
5. Works over car Bluetooth (speaker + mic routing).
6. When the call ends, Ashley remembers what was discussed.
7. Temporary signal loss feels like an interruption, not a call death.

ARCHITECTURE OVERVIEW
---------------------

Current flow (push-to-talk, sequential):
  [Record to file] → [POST base64] → [Deepgram STT] → [Claude text] →
  [ElevenLabs TTS] → [Play file]
  Latency: 2-4 seconds per turn. Requires button hold.

Target flow (phone call, streaming):
  [500ms audio blobs] → [WebSocket] → [Deepgram streaming STT]
       ↓ (on speech_final event)
  [Claude Sonnet with full Ashley context]
       ↓
  [ElevenLabs eleven_turbo_v2_5 TTS streaming]
       ↓
  [Stream audio chunks back] → [Play continuously] → [Mic reopens automatically]
  Latency: ~800ms-1.5s per turn. No button required after call starts.

NOTE ON MOBILE AUDIO: expo-audio 1.1.1 does not support raw PCM chunk
streaming on Android. The primary mobile audio strategy is 500ms segmented
blob loop (see Phase 2). Do not plan around PCM streaming.

PHASE 1: SERVER WEBSOCKET INFRASTRUCTURE
--------------------------------------------------------------------------------

FILES TO CREATE/MODIFY:
  artifacts/api-server/src/index.ts              (MODIFY — capture server ref)
  artifacts/api-server/src/routes/voice-call.ts  (NEW)
  artifacts/api-server/package.json              (add 'ws' package)

NOTE: WebSocket server attaches to index.ts, NOT app.ts.
  app.ts creates the Express app and exports it.
  index.ts calls app.listen() — this is where ws.Server must be attached.

  Required change in index.ts:
    BEFORE: app.listen(port, callback)
    AFTER:  const server = app.listen(port, callback)
            // then: wsServer.attach(server)

Step 1.1 — Add WebSocket support
  - Install 'ws' package
  - In index.ts: capture http.Server from app.listen()
  - Create ws.Server attached to that server reference
  - Mount under /api/voice/call path

Step 1.2 — Connection authentication
  CORRECTED AUTH PATTERN (confirmed against live codebase):

  The WebSocket handshake must send TWO headers:
    Authorization: Bearer <API_AUTH_KEY value>
    X-Device-Id: <device-uuid>

  Server auth files:
    artifacts/api-server/src/middleware/auth.ts     — validates Bearer token
    artifacts/api-server/src/middleware/deviceId.ts — reads X-Device-Id

  IMPORTANT: The env var the server reads is API_AUTH_KEY, not API_SECRET.
  Confirm API_AUTH_KEY is set on the API Server artifact before Phase 1.
  If only API_SECRET is set and the server reads API_AUTH_KEY, all
  authenticated routes will return 500.

  Do NOT send X-API-Key header — the server does not check this header name.

Step 1.3 — Session initialisation
  On authenticated connection, assign a sessionId and connectionGeneration:
  a) Load from database: Ashley's profile, recent memories (last 20),
     conversation summary for this device, recent message history
  b) Build the call system prompt (see Phase 4 for prompt spec)
  c) Open a Deepgram streaming STT WebSocket connection
  d) Confirm session ready to client with a "call_connected" message
     including sessionId and connectionGeneration

Step 1.4 — Audio routing loop (the core loop)
  CLIENT → SERVER:
    Binary WebSocket frames: 500ms audio blobs (primary strategy)
    All control messages include: sessionId, connectionGeneration,
    sequenceNumber, timestamp (see F12 — Stale Connection Protection)

  SERVER processing:
    a) Validate sessionId and connectionGeneration — reject stale messages
    b) Forward audio frames to Deepgram streaming connection
    c) Deepgram fires transcript events (is_final=false for partials)
    d) On speech_final=true: assign utteranceId, complete utterance ready
    e) Validate utteranceId not already processed (dedup protection)
    f) Send utterance to Claude Sonnet — assign turnId and responseId
       (use streamChatText() with forceProvider: "anthropic")
       (add AbortSignal with 30s ceiling for call responsiveness)
    g) Validate turnId/responseId still current before sending to TTS
    h) Send Claude response to ElevenLabs streaming endpoint — assign speechId
    i) Stream ElevenLabs audio chunks back as binary WS frames
       (discard any audio from cancelled speechIds)
    j) Send "tts_done" signal when audio stream ends
    k) Mic reopens on client side after "tts_done"

  Interruption handling:
    a) Client sends "interrupt" signal if user starts speaking mid-TTS
    b) Server sends "interrupt_ack" immediately so UI switches to listening
    c) Server cancels current TTS stream (marks speechId as cancelled)
    d) Server cancels Claude AbortSignal if in flight
    e) New utterance processing begins with new turnId/responseId

Step 1.5 — Session end
  On WebSocket close (either side):
  a) Close Deepgram connection
  b) Filter transcript before saving — exclude connection tests, failed
     turns, and accidental audio (see F18 — Memory Contamination Filter)
  c) Write clean transcript to conversation history with source="voice_call"
     NOTE: Confirm source column exists in @workspace/db schema first.
     May require manual ALTER TABLE SQL (drizzle-kit generate is broken).
  d) Run conversation summary rollup if session > threshold length

Step 1.6 — Session recovery (F11)
  A temporary signal loss should feel like an interruption, not call death.

  SERVER behaviour on disconnect:
    - Keep session alive for 60 seconds after WebSocket closes
    - Preserve: transcript, context, Deepgram connection state
    - Allow the same X-Device-Id to reclaim the session within 60 seconds
    - After 60 seconds with no reconnect: close Deepgram, save transcript,
      end session cleanly

  CLIENT reconnect strategy:
    - 3 immediate reconnect attempts (no delay)
    - Wait 5 seconds
    - Repeat for 5 rounds
    - Maximum 15 recovery attempts total
    - After exhausting attempts: show "Call ended" UI, do not auto-restore

  TTS rule on reconnect:
    - Do NOT replay half-finished speech after reconnect
    - Resume naturally or Ashley asks "did you catch that?"
    - Never resume from mid-sentence on a new connection

  Cold restart policy (F20):
    - After a full app process death (not just disconnect): do NOT
      automatically restore the call
    - Require user to manually start a new call
    - Rationale: user may not want to continue; process death is a
      hard boundary, not a soft disconnect

  DEEPGRAM_API_KEY                    — CONFIRMED PRESENT
  ELEVENLABS_API_KEY                  — CONFIRMED PRESENT
  ELEVENLABS_VOICE_ID                 — CONFIRMED PRESENT (Daisy)

PHASE 2: MOBILE STREAMING AUDIO PIPELINE
--------------------------------------------------------------------------------

Files to create/modify:
  artifacts/mobile/lib/voiceCall.ts               (new — core call manager)
  artifacts/mobile/lib/voiceCallAudio.ts          (new — audio capture/playback)
  artifacts/mobile/app/voice-call.tsx             (new — call screen)

Step 2.1 — WebSocket client (voiceCall.ts)
  - Connect to wss://[production-domain]/api/voice/call
  - Send auth headers in connection request:
      Authorization: Bearer <API_AUTH_KEY value>
      X-Device-Id: <device-uuid>
  - All messages include: sessionId, connectionGeneration, sequenceNumber,
    timestamp
  - Handle: call_connected, binary audio frames, tts_done, interrupt_ack,
    error
  - Implement reconnect strategy (see F11 — Session Recovery)
  - Do NOT auto-restore after cold app restart
  - Expose: startCall(), endCall(), sendAudioChunk(), onAshleySpeaking,
            onAshleyFinished, onTranscript

Step 2.2 — Mobile audio capture (voiceCallAudio.ts)
  PRIMARY STRATEGY: 500ms segmented blob loop.

  expo-audio 1.1.1 does NOT support raw PCM chunk streaming on Android.
  Do not attempt PCM streaming as a primary path.

  Implementation:
    while (callActive) {
      startRecording()
      await sleep(500)
      blob = stopRecording()
      sendBinaryFrame(blob)
    }

  This produces ~600-800ms STT latency per chunk.
  Acceptable for conversation flow.

  Audio backpressure (F16):
    - Monitor outgoing audio queue size
    - If queue exceeds 2 seconds of audio, drop oldest chunks
    - Ashley must never speak several seconds behind the conversation
    - Log backpressure events for debugging

Step 2.3 — Streaming audio playback
  - Receive binary WebSocket frames (ElevenLabs MP3 output chunks)
  - Track speechId — discard chunks from cancelled speechIds
  - Buffer and play continuously using expo-audio
  - Must handle: mid-stream start (don't wait for full audio)
  - Must handle: stream interruption (stop playback cleanly on interrupt_ack)

Step 2.4 — Android audio session management
  This is the hardest part. Requirements:
  a) While Ashley is speaking: speaker active, mic monitoring (not sending)
  b) After tts_done: mic starts sending, speaker on standby
  c) Must handle Bluetooth audio routing (HFP profile for car)
  d) Must request appropriate audio focus (AUDIOFOCUS_GAIN for calls)
  e) Must release focus correctly so other apps resume after call ends

  Android-specific gotchas:
  - STREAM_VOICE_CALL vs STREAM_MUSIC affects Bluetooth routing
  - Samsung S24 Ultra may require explicit setSpeakerphoneOn() calls
  - Background audio requires a Foreground Service (notification required)

  Silence lifecycle (F17):
    - After 30 seconds of silence: Ashley says "still there?"
    - After 60 seconds of silence: Ashley says "I'll let you go — call me back"
    - After 90 seconds of silence: server closes session cleanly
    - Do not leave open connections consuming API costs indefinitely

Step 2.5 — Foreground Service for background operation
  CONFIRMED STATUS: expo-task-manager NOT installed. EAS dev build required.

  - Foreground service is required for screen-off car use
  - Cannot be implemented until expo-task-manager is installed
  - Installation requires a new EAS build cycle (new APK + install)
  - Does NOT block Phases 1 or 2 — screen-on operation works without it
  - Plan as Phase 3 prerequisite — factor EAS build time into timeline

PHASE 4: ASHLEY CONTEXT + POST-CALL MEMORY
--------------------------------------------------------------------------------

Call system prompt spec:
  "You are Ashley. [Full persona description]. [Relationship history summary].
   [Recent memories]. [Current context: time, location if available].

   We are having a voice call right now. Keep your replies natural and
   conversational — short to medium length, no bullet points, no markdown,
   no stage directions. Speak as you would in a real phone call. You can
   ask follow-up questions. You can be playful or serious as the moment
   warrants. This is real-time — respond at conversation pace."

Post-call memory (F18 — Memory Contamination Filter):
  Before saving, filter the transcript:
    - Exclude: connection test messages, system events, failed turns
    - Exclude: accidental audio (less than 1 second, no transcript content)
    - Exclude: turns where Deepgram returned empty string
    - Include: only genuine conversation exchanges

  Save clean transcript to messages table with source="voice_call".
  Roll up into conversation summary by existing summariser.
  Next chat session: Ashley has memory of call content.

Voice-appropriate response tuning:
  - No markdown (strip asterisks, hyphens, pound signs before TTS)
  - Shorter average response length (~2-4 sentences)
  - More natural verbal acknowledgement ("yeah", "right", "I get that")
  - Can trail off / be interrupted mid-sentence naturally

CONTENT POLICY IN VOICE CALLS — NSFW / MATURE MODE
================================================================================

SHORT ANSWER: voice calls inherit the EXACT same three-layer content policy as
text chat. No special voice content mode, no downgrade, no separate consent.
If the user has Mature Mode enabled in text chat, it is enabled in the call.

Layer 1 — Provider Floor (immutable):
  No sexual content involving minors. No non-consensual scenarios.
  Cannot be turned off by any mode, setting, or instruction.

Layer 2 — Content Mode ("standard" or "mature"):
  Standard: default. Intimacy ceiling at level 3.
  Mature: unlocked when ASHLEY_MATURE_MODE_AVAILABLE=true AND user has
          confirmed 18+ via POST /profile/confirm-adult.
          Intimacy ceiling rises to level 5.

Layer 3 — Intimacy Level (0-5 ladder).

HOW THIS FLOWS INTO A VOICE CALL SESSION
------------------------------------------
At session start (Step 1.3), the server:
  a) Loads user profile by device-id
  b) Calls getPolicyFor(profile) → ResolvedPolicy
  c) Builds system prompt using:

  import { getPolicyFor, buildProviderFloorBlock, buildModeBlock,
           buildIntimacyBlock } from "../lib/contentPolicy";

  const policy = getPolicyFor(profile);
  const systemPrompt = [
    ASHLEY_CORE_SPEC,
    buildProviderFloorBlock(),
    buildModeBlock(policy),
    buildIntimacyBlock(policy),
    buildVoicePresenceFloor(),          // Stage 5 — write in contentPolicy.ts
    VOICE_CALL_BEHAVIOUR_BLOCK,
  ].join("\n\n");

VOICE-PRESENCE SAFETY FLOOR (Stage 5, not yet built):
  "Voice presence informs WARMTH, never DIAGNOSIS."
  Allowed: "you sound tired tonight"
  Forbidden: "you sound depressed", "you're having a panic attack"
  If she notices something concerning, she invites — never narrates.
