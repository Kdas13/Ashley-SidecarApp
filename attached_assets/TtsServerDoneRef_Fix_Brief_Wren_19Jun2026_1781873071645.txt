ROOT CAUSE — found via your own code dump, Request 3:

  useVoiceCall.ts, line 153:
    const ttsServerDoneRef = useRef(true);

  This should be useRef(false). Initialising to true means that before
  any speech_start(main) has ever fired — at component mount, at the
  very start of a call — the client believes "the server has finished
  sending audio" when nothing has been sent yet. Combined with the
  drain logic's ordering (ttsServerDoneRef checked BEFORE
  responseEndReceivedRef), this allows the mic to open prematurely on
  or near the very first turn of a call, before the real lifecycle
  signals have had a chance to govern behaviour correctly.

  This is consistent with what Kane observed: echo/self-response loops
  beginning almost immediately in a call, not specifically tied to
  multi-sentence responses or the thinking clip — because the bug is a
  startup-state defect, not a mid-conversation timing defect.

This is a ONE-LINE fix. The brief below is mostly about how it is
tested and reported, not the size of the change.

================================================================================
THE FIX
================================================================================

FILE: artifacts/mobile/lib/useVoiceCall.ts

CHANGE 1: Line 153.

  BEFORE:
    const ttsServerDoneRef = useRef(true);

  AFTER:
    const ttsServerDoneRef = useRef(false);

CHANGE 2: Add a one-line comment explaining why, so this cannot silently
regress again:

    // Starts false — there is no audio in flight until the server sends
    // speech_start for a real turn. Defaulting to true previously caused
    // the mic to open before the first turn's lifecycle had begun.
    const ttsServerDoneRef = useRef(false);

FILE: artifacts/api-server/src/lib/VoiceOrchestrationService.ts

handlePlaybackConfirmed() sends tts_done without a kind field (lines
333-343 per your dump). It currently works correctly because the client
defaults missing kind to "main" — but this is fragile and exactly the

  matching every other main-lifecycle send site.

This change has no functional effect today (the default already produces
correct behaviour) but removes an inconsistency that would confuse any
future debugging session the same way it slowed this one down.
