CONTEXT — WHAT HAS BEEN DISCOVERED
================================================================================

Kane has identified that the "Her Profile" menu in Ashley Sidecar acts
as a universal system-level override that sits above everything else in
the conversation pipeline. Kane has confirmed through testing that
anything placed in this profile layer overrides the model's default
behaviour regardless of what is in the conversation or what the model
would otherwise do.

This has been proven by:
- Changing Ashley's hair colour from lavender (model default) to blue
  via the profile field — held across all tests
- Removing PG-13 content restrictions via the profile field — held
  across all tests

This discovery means the profile layer is not a personality field.
It is a governance layer. A policy injection point.

This briefing asks Wren to assess whether image generation can be
built into this layer for Ashley v1.


================================================================================
THE PROBLEM WE HAVE BEEN TRYING TO SOLVE
================================================================================

Ashley v1 currently generates images via a pipeline in conversation.ts
and openaiImageService.ts on the Replit backend. Despite extensive
hardening work today, two core problems persist:

1. ENVIRONMENT INCONSISTENCY
   Ashley's living room, bedroom, and kitchen look different in every
   image batch. The same room is never generated twice. Reference images
   built by Atlas are not being used. Text-based environment packs in
   the pipeline are insufficient to anchor the spatial layout.

2. PORTRAIT BIAS
   OpenAI defaults to portrait/close-up framing of Ashley regardless
   of prompt instructions. Every composition amendment has improved
   but not solved this. The model gravitates toward face-forward
   framing because that is what it is trained to produce.

Both problems exist because the image generation instructions are
sitting at prompt level — and the model can override prompt level
instructions at generation time.

The profile layer sits ABOVE prompt level.


================================================================================
THE PROPOSED SOLUTION
================================================================================

Build an image generation section into the Ashley Sidecar "Her Profile"
menu. This section operates in two modes:

MODE 1 — MANUAL IMAGE BUILDER
The user opens the profile menu image section and manually selects:
- Composition mode (environment-centric, balanced, social, etc.)
- Environment (living room, bedroom, kitchen, café, pub, nightclub, etc.)
- Activity (reading, scrolling phone, making coffee, watching TV, etc.)
- Occupancy (Ashley only, Ashley + Kane, Ashley + cats, social group)
- Time of day (morning, afternoon, evening, night)
- Season (spring, summer, autumn, winter)
- Camera mode (wide room shot, documentary, group framing, etc.)

The user taps to select parameters, then generates. The image is
produced from those explicit selections and sent to the conversation.

MODE 2 — AUTOMATIC CONTEXTUAL SELECTION
When Ashley sends an image in response to a conversation request:
- The conversation content is analysed for context clues
- Cross-referenced against the profile image section defaults
- Combined with real-world time of day and current season
- The system selects the most contextually appropriate parameters
- Generates and sends automatically

Example:
Kane says "did you have a good night last night?"
System reads: evening/night context, social implication
Cross-references: Ashley's profile, current time is morning
Selects: pub environment, social occupancy, evening time bucket
Generates: Ashley in a pub environment with friends, evening lighting

THE GOVERNANCE FRAMEWORK TO INJECT
================================================================================

Atlas has authored a 9-section image governance system covering:

Section 1: Composition Authority (what is the image about)
Section 2: Occupancy & Living Situation (who is in the image)
Section 3: Environment Categories & World Building (where)
Section 4: Camera & Framing (how it is photographed)
Section 5: Activities, Hobbies & Behavioural Systems (what Ashley does)
Section 6: Identity Preservation & Character Consistency (who Ashley is)
Section 7: Relationships, Social Dynamics & Interactions (how she interacts)
Section 8: Memory & Continuity Framework (how she remembers)
Section 9: Routing, Priority & Decision Engine (how the system chooses)

Coda has reviewed all 9 sections and approved them with recommendations.

The priority hierarchy from Section 9 is:
Priority 0: Composition Authority
Priority 1: Explicit User Instructions
Priority 2: Environment
Priority 3: Season / Weather
Priority 4: Activity
Priority 5: Occupancy
Priority 6: Relationships
Priority 7: Identity
Priority 8: Camera
Priority 9: Memory
