SCOPE DECISION (recorded)
--------------------------
Mid-call checkpointing was removed after cost analysis. £100/month buys
hundreds of hours of voice call time — Kane cannot realistically reach that
ceiling. Mid-call summaries solve a problem that doesn't exist and add
unnecessary complexity. End-of-call summarisation is retained in full.
TokenBudgetMonitor is removed entirely. is_mid_call and checkpoint_turn_id
columns are removed from schema.
summarisation_jobs recovery table is RETAINED — the fire-and-forget recovery
path is still valuable if the process dies immediately after call end.

================================================================================
OVERVIEW
--------
When a voice call ends, fire an asynchronous summarisation job. The job
generates a structured LLM summary and writes it to PostgreSQL. The call
pipeline must never block on this. A job record is persisted before the async
task runs so recovery is possible if the process dies mid-flight.

--- TABLE: call_summaries ---

CREATE TABLE call_summaries (
  id                TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
  session_id        TEXT NOT NULL,
  version           INTEGER NOT NULL DEFAULT 1,
  status            TEXT NOT NULL DEFAULT 'pending_review'
                      CHECK (status IN ('committed', 'pending_review', 'superseded')),
  topic             TEXT,
  tone              TEXT,
  open_items        JSONB NOT NULL DEFAULT '[]'::jsonb,
  summary_text      TEXT NOT NULL,
  confidence_score  INTEGER NOT NULL CHECK (confidence_score >= 0 AND confidence_score <= 100),
  generated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  committed_at      TIMESTAMPTZ,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

--- TABLE: call_decisions ---

CREATE TABLE call_decisions (
  id             TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
  session_id     TEXT NOT NULL,
  summary_id     TEXT NOT NULL REFERENCES call_summaries(id) ON DELETE CASCADE,
  turn_id        TEXT NOT NULL,
  decision_type  TEXT NOT NULL
                   CHECK (decision_type IN ('action', 'commitment', 'preference', 'question')),
  description    TEXT NOT NULL,
  resolved       BOOLEAN NOT NULL DEFAULT false,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

--- TABLE: summarisation_jobs (recovery table) ---

CREATE TABLE summarisation_jobs (
  id          TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
  session_id  TEXT NOT NULL,
  status      TEXT NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending', 'running', 'complete', 'failed')),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CLASS: CallSummarisationService

PUBLIC METHOD: summariseCall(sessionId: string, turns: ConversationTurn[]): void

  - Fire-and-forget. Do NOT return a Promise. Do NOT await this method anywhere.
  - Wrap the entire async body in a self-invoking async IIFE.
  - Catch all errors internally. Log errors but never throw.

  INTERNAL ASYNC BODY — in order:

  STEP 1: Write job record to summarisation_jobs table BEFORE starting async work.
    - status: 'pending'
    - If this write fails, log and return. Do not proceed.

  STEP 2: Update job status to 'running'.

  STEP 3: Transcript preparation.
    - Filter turns: only include role === 'user' or role === 'assistant'.
      Exclude system messages entirely.
    - Cap: slice to LAST 150 turns maximum before passing to LLM.

  STEP 4: Short call bypass.
    - If filtered turns.length < 4:
      - Skip keyword validation and keyword scoring penalties entirely.
      - Set confidence_score = 100.
      - Proceed directly to LLM call with score pre-set.

  STEP 5: LLM summarisation call.
    - Call callLLM() with a prompt instructing it to return ONLY valid JSON.
    - No preamble, no markdown fences.
    - Required JSON shape:
      {
        "topic": string,
        "tone": string,
        "open_items": string[],
        "summary_text": string,
        "confidence_score": number,   // 0-100
        "decisions": [
          {
            "turn_id": string,
            "decision_type": "action" | "commitment" | "preference" | "question",
            "description": string,
            "resolved": boolean
          }
        ]
      }

  STEP 6: Parse and validate LLM response.
    - Strip any accidental markdown fences before JSON.parse().
    - Perform STRICT runtime type mapping on every field. Do not silently cast.
      If a field is wrong type or missing, log the field name and either use a
      safe default or abort the write with a logged error.
    - Clamp confidence_score: Math.max(0, Math.min(100, score))
      Apply BOTH floor and ceiling.

  STEP 7: writeSummary() — run inside db.transaction().
    INSIDE TRANSACTION:
      a. Supersede ALL existing rows for this session_id where status is
         'committed' OR 'pending_review'. Set status = 'superseded' for both.
      b. Insert new row into call_summaries with status = 'pending_review'.
      c. Insert all decision rows into call_decisions, FK → new summary id.
    END TRANSACTION.

  STEP 8: Update job status to 'complete'. On any catch: update to 'failed'.
