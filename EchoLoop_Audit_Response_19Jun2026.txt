ITEM 4 — ANALYSIS FROM ITEMS 1–3 ONLY
================================================================================

The question: "why would response_end for the MAIN turn not reach the client
before 15 seconds elapses on a normal-length Gemini response?"

First, a correction required by the code: the 15-second safety timeout (Item 1,
lines 670–691) does NOT send any message to the client. It resets server-side
state only. So the 15-second timeout is not a mechanism that puts
ttsServerDoneRef=true on the client. The comment in Item 3 at line 197:
  "tts_done already received (server safety timeout path)"
is misleading. There is no server-side 15s path that delivers tts_done to the
client. The server's 15s timeout is purely a server-state reset.

What the code in Items 1–3 does show:

MECHANISM A — gate check at line 646 in Item 1:
  response_end is only sent if: session.currentSpeechId === speechId

  session.currentSpeechId is set to null by the hard budget timeout at line 568:
    session.currentSpeechId = null;
  The hard budget fires at MAX_TURN_BUDGET_MS = 45,000 ms (line 571, constant
  at line 35 of the file).

  If the hard budget fires, session.currentSpeechId becomes null before the LLM
  stream exits. The gate at line 646 then fails and response_end is NEVER sent.
  The hard budget DOES send tts_done(main) at line 554–566.

  Result on client: ttsServerDoneRef=true (from budget tts_done), but
  responseEndReceivedRef=false (response_end was never sent).

  On a normal one-sentence Gemini response ("Hey, Ashley.") the hard budget at
  45s should NOT fire. There is no other mechanism in Items 1–3 that nullifies
  session.currentSpeechId mid-turn except the hard budget and interrupt paths.

MECHANISM B — ordering of checks in playNext (Item 3, lines 196–210):
  The drain branch is an if / else-if / else tree:

    if (ttsServerDoneRef.current) {
      // open mic immediately — does not check responseEndReceivedRef
    } else if (responseEndReceivedRef.current) {
      // send playback_confirmed
    } else {
      // inter-sentence gap — wait
    }

  ttsServerDoneRef is checked FIRST. If it is true at drain time, the mic opens
  regardless of responseEndReceivedRef. The drain that was logged:
    "playNext: drained responseEnd=false srvDone=true phase=speaking"
  took the first branch, which means ttsServerDoneRef=true and
  responseEndReceivedRef=false simultaneously at that moment.

MECHANISM C — responseEndReceivedRef reset (speech_start handler, not in
  Items 1–3 — in the WS message handler): responseEndReceivedRef is reset to
  false by speech_start(main). If a new turn's speech_start arrives between
  response_end being received and tts_done arriving, the drain would see
  responseEnd=false, srvDone=true — exactly the observed log.

What Items 1–3 alone CANNOT show:
  Why the very FIRST mic-open goes through this path on a one-sentence response
  where neither the 45s hard budget nor an interrupt should fire. The
  playback_confirmed handler (which sends tts_done in the normal success path)
  is not in any of Items 1–3, so the timing of when tts_done is sent relative
  to response_end cannot be determined from these three sections alone.

  Items 1–3 show the vulnerability (the drain opens mic on ttsServerDoneRef
  without requiring responseEndReceivedRef) but do not show the initial
  trigger.
