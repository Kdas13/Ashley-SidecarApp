STEP M2 — PCM amplitude metering
[FILE]  artifacts/mobile/lib/voiceInput.ts  ~line 86

IN:     Raw PCM chunk from AudioRecord 'data' event (base64 string, ~100ms).

PROCESSING:
  1. atob(chunk) → binary string of 16-bit LE samples
  2. Iterate pairs of bytes: byte[i] | (byte[i+1] << 8) → unsigned int16
  3. If > 32767: subtract 65536 → signed int16
  4. Track max absolute value across all samples in this chunk
  5. dBFS = max > 0 ? 20 * log10(max / 32768) : -160

OUT:    meteringRef.current = dBFS (range: -160 to ~0)
        setMetering(dBFS) → triggers React state update → VAD effect fires

NOTE:   -160 dBFS is a floor sentinel, produced only when every sample is
        exactly zero (true hardware silence or mic not yet producing data).
        Computed from raw PCM buffer — immune to Samsung DSP/AGC suppression
        of getMaxAmplitude().

STEP M3 — VAD loop (client-side silence detection)
[FILE]  artifacts/mobile/lib/useVoiceCall.ts  ~line 592

Fires on every change to recorder.metering (i.e. every ~100ms PCM chunk).
Thresholds (all tunable at top of file):
  SILENCE_DB    = -28 dBFS
  SILENCE_MS    = 3500ms  (standard silence tolerance)
  SILENCE_MS_SHORT = 4500ms  (for short segments < 1000ms speech)
  SILENCE_THRESHOLD_SHORT_MS = 1000ms
  MIN_SPEECH_MS = 500ms  (minimum speech duration to submit)
  VAD_DELAY_MS  = 500ms  (dead zone after mic open)

GATE:   vadActiveRef.current === false → return immediately (dead zone, or
        Ashley is speaking).
GATE:   db === null → probe counter incremented, return immediately.
        null metering = device does not support PCM amplitude reporting.
        After 2s probe window closes: if null frames dominate → vadMode="blind",
        autoSubmit fallback takes over (4s timer, not silence detection).

ON db > SILENCE_DB (speech detected):
  - isUserSpeakingRef = true
  - speechStartAtRef = now (if transitioning from silence)
  - phase set to "user_speaking"
  - lastSpeechAtRef = now
  - segmentSpeechMsRef = now - speechStartAtRef
  - autoSubmit fallback timer RESET on every detected sound (so it only
    fires after 4s of total silence from the LAST detected sound)

ON db ≤ SILENCE_DB (silence):
  - If isUserSpeakingRef was true (speech → silence transition):
    - Compute silenceDuration = now - lastSpeechAtRef
    - Select threshold:
        segmentSpeechMsRef < SILENCE_THRESHOLD_SHORT_MS → use SILENCE_MS_SHORT
        otherwise → use activeSilenceMsRef (default SILENCE_MS = 3500ms,
        may have been overridden to 800ms by server set_silence_threshold)
    - If silenceDuration >= threshold → submitSegment()

SPECIAL CASE — db = -160 dBFS:
  -160 < SILENCE_DB (-28), so it is treated identically to any other silence
  value. If the user was previously speaking (isUserSpeakingRef=true), the
  silence counter runs down from lastSpeechAtRef. After activeSilenceMsRef,
  submitSegment() fires. If the user was never speaking, nothing is submitted
  from VAD. The server-side silence monitor fires independently at 30s/60s/90s.

TIMER:  autoSubmit (blind mode only) — 4s from last detected sound (or mic
        open if no sounds). Sets isBlindCaptureRef=true, calls submitSegment().

SECTION 4 — SERVER-SIDE SILENCE MONITOR (INDEPENDENT TIMER)
================================================================================

[FILE]  artifacts/api-server/src/routes/voice-call.ts
        startSilenceMonitor()  ~line 266

Started once per session from index.ts after call_connected is sent.
Runs on its own 5-second poll interval (SILENCE_POLL_MS = 5000ms).
The clock is: Date.now() - (session.lastAudioReceivedAt ?? session.callStartTime)

TICK LOGIC:

  GATE: state === "closed" / "failed" / "closing" → stop polling, return.
  GATE: state NOT IN ("listening", "active") → skip measurement this tick
        (don't count silence while LLM or TTS is active).
  GATE: session.awaitingPlaybackConfirm → skip measurement this tick.

  Measure silenceMs = elapsed since lastAudioReceivedAt (or callStartTime).

  silenceMs >= SILENCE_FORCE_MS (90s):
    WS.send({ type: "call_ended", reason: "silence_timeout" })
    registry.finalise("silence_timeout_failsafe")
    Poll stops. No TTS.

  silenceMs >= SILENCE_END_MS (60s) AND silenceWarningSent=true:
    speakFarewell(session):
      speakFallback(session, "I'll let you go — call me back when you're ready.")
      WS.send({ type: "call_ended", reason: "silence_timeout" })
      registry.finalise("silence_timeout")
    Poll stops.

  silenceMs >= SILENCE_WARN_MS (30s) AND !silenceWarningSent:
    session.silenceWarningSent = true
    speakFallback(session, "Still there?")

  Schedule next tick: setTimeout(tick, SILENCE_POLL_MS = 5000ms)

────────────────────────────────────────────────────────────────────────────────
CRITICAL NOTE — SILENCE MONITOR BYPASSES filterForbiddenPhrases

  speakFallback() in voice-call.ts calls streamSpeechElevenLabs() DIRECTLY.
  It does NOT go through flushSentenceToTTS() in VoiceOrchestrationService.ts.
  Therefore filterForbiddenPhrases() is NOT applied to these strings.

  "Still there?"  — matches /still there/i (FORBIDDEN_PHRASE_PATTERNS[11])
                    but WILL be spoken by the 30s silence monitor.
  "I'll let you go — call me back when you're ready."
                  — matches /let you go/i (FORBIDDEN_PHRASE_PATTERNS[1])
                    but WILL be spoken by the 60s silence monitor.

  These are hardcoded strings in a different function (speakFallback) that
  predates the filter. The filter only covers the LLM output path.

SECTION 5 — WHAT HAPPENS AT -160 dBFS
================================================================================

Trigger: AudioRecord.on("data") fires with a chunk where every sample = 0,
producing: dBFS = 20 * log10(0 / 32768) → handled specially as -160.

Step-by-step when mic is in listening state and -160 arrives:

  1. voiceInput.ts:  meteringRef = -160, setMetering(-160)
  2. useVoiceCall.ts VAD effect fires:
       db === null? No (-160 is not null). vadProbeUsableRef++.
       meteringEverReceivedRef = true.
       vadActiveRef.current? If still in 500ms delay → return.
       db (-160) > SILENCE_DB (-28)? NO → silence branch.
  3. isUserSpeakingRef?
       YES (was speaking): no transition needed; isUserSpeakingRef stays true.
         Check silence duration: now - lastSpeechAtRef.
         If silence duration >= activeSilenceMsRef → submitSegment() fires.
       NO (never spoken this segment): no change. VAD loop does nothing.
         autoSubmit timer (blind mode only) counts down independently.
         Server-side silence monitor counts down independently.
  4. If submitSegment() fires: same flow as Step M4 above.
     STT hallucination guard: segmentSpeechMsRef likely < MIN_STT_SPEECH_MS
     if speech was very brief → segment dropped.
  5. If submitSegment() does NOT fire (was never speaking):
     Server-side silence monitor polls every 5s against lastAudioReceivedAt.
     At 30s: "Still there?" (speakFallback — bypasses filter, see Section 4)
     At 60s: farewell + call_ended
     At 90s: force close

SEQUENCE WHEN USER GOES COMPLETELY SILENT FROM THE START:
  T+0s:    mic open, VAD delay starts
  T+0.5s:  VAD active, -160 readings every ~100ms, no submission (never spoke)
  T+5s:    silence monitor first tick: silenceMs ~5s, no action
  T+30s:   silence monitor tick: silenceMs >= 30s, silenceWarningSent=false
             → speakFallback("Still there?") → state changes to tts_streaming
               → monitor skips measurement while tts/playback active
             → silenceWarningSent = true
  T+60s:   silence monitor tick: silenceMs >= 60s, silenceWarningSent=true
             → speakFarewell() → "I'll let you go..." + call_ended + finalise
  T+90s:   failsafe (only reached if 60s path failed):
             → direct call_ended + finalise, no TTS
