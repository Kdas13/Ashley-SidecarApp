CONFIRMATION 2 — AUTHENTICATION PATTERN
STATUS: CHANGED from what the protocol document described

Actual pattern (three-piece, not two):

  Authorization: Bearer <token>
    Server middleware (requireApiKey in middleware/auth.ts) validates this
    against env var API_AUTH_KEY (NOT API_SECRET and NOT X-API-Key header).

  X-Device-Id: <uuid>
    Separate header, read by requireDeviceId in middleware/deviceId.ts,
    stored as req.deviceId.

  X-API-Key: <key>
    Mobile client sends this (from EXPO_PUBLIC_API_KEY) but the server
    middleware does NOT check this header name — it checks Authorization Bearer.

Auth files:
  artifacts/api-server/src/middleware/auth.ts
  artifacts/api-server/src/middleware/deviceId.ts

FLAG 1: The protocol document said "X-API-Key + Authorization: Bearer device-id"
— this is wrong on two counts. The Bearer token is the API key (not the device
ID), and the device ID travels on X-Device-Id, not Authorization.

The WebSocket handshake must send:
  Authorization: Bearer <API_AUTH_KEY_VALUE>
  X-Device-Id: <device-uuid>

FLAG 2: The server checks API_AUTH_KEY env var. replit.md documents API_SECRET.
These may be aliases or the naming drifted during migration. The env check
shows API_SECRET: MISSING. Confirm which env var is actually set on the server
before Phase 1. If only API_SECRET is set and the server reads API_AUTH_KEY,
all authenticated routes will return 500.

CONFIRMATION 7 — EXPO AUDIO CAPABILITIES
STATUS: CONFIRMED — file-based only, no PCM streaming

- Expo SDK: ~54.0.27 — CONFIRMED
- expo-audio: ~1.1.1 — CONFIRMED, installed
- Recording method: useAudioRecorder(RecordingPresets.HIGH_QUALITY) records
  to a file (m4a/wav/caf depending on platform), then FileSystem.readAsStringAsync
  converts to base64. There is NO raw PCM chunk streaming capability.
- expo-task-manager: NOT installed
- expo-background-fetch: NOT installed

Audio permissions configured:
  iOS: NSMicrophoneUsageDescription set in app.json
  Android: RECORD_AUDIO permission set in app.json
  expo-audio plugin: configured with microphonePermission string

Existing audio lib files in artifacts/mobile/lib/:
  voiceInput.ts     — push-to-talk recording (file-based)
  voiceOutput.ts    — TTS playback (expo-audio player)
  voiceActivity.ts  — activity monitoring
  audioState.ts     — shared audio state/logging
  useSpeakMessage.ts — speak-on-new-message hook

FLAG (high risk): There is no raw PCM streaming path in expo-audio 1.1.1.

Recommended approach: use the 500ms segmented blob loop as the PRIMARY
implementation plan, not a fallback. Implementation:

  while (callActive) {
    startRecording()
    await sleep(500)
    blob = stopRecording()
    sendBinaryFrame(blob)
  }

This produces ~600-800ms STT latency per chunk vs ~100ms for true PCM
streaming. Acceptable for a conversation flow. Avoids the risk of finding
PCM streaming unsupported on the S24 Ultra mid-build.

If expo-audio SDK 54 exposes onAudioSampleReceived or equivalent, true PCM
streaming can be explored in Phase 2, but do not plan around it.
