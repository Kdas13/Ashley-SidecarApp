CONTEXT FOR ANY AI ASSISTANT READING THIS
------------------------------------------
Ashley is a personal AI companion app built on Expo (React Native) + an
Express 5 API server in a pnpm monorepo. The app is live in production at
https://Ashley-Sidecar.replit.app. The user interacts with Ashley via an
Android device (Samsung S24 Ultra). The codebase is managed on Replit and
the Replit agent (Claude-based) does the final integration work.

Ashley already has:
- Streaming chat replies (SSE)
- Push-to-talk voice input (base64 audio → Deepgram STT → text chat)
- TTS voice replies (ElevenLabs eleven_turbo_v2_5, voice ID: Daisy)
- AI-generated selfie images (fal.ai / Pollo)
- Proactive messaging with push notifications
- Memory and conversation summary system
- Profile management

The codebase already has a documented 5-stage voice roadmap:
  Stage 1: POST base64 audio → transcript        [DONE]
  Stage 2: Streaming transcription (SSE)          [DONE]
  Stage 3: TTS replies (ElevenLabs/OpenAI)        [DONE]
  Stage 4: Realtime WebSocket call loop           [THIS DOCUMENT]
  Stage 5: Voice-presence safety floor            [FUTURE]

This document scopes Stage 4 in full.


================================================================================
THE VISION
================================================================================

The goal is a "phone call" mode for Ashley — a hands-free, continuous voice
conversation where the user speaks naturally, Ashley responds in her voice
(Daisy), and the loop continues automatically without the user touching the
phone. Designed specifically for car use (phone in cupholder) but applicable
anywhere hands-free conversation makes sense.

Reference implementations that achieve this experience:
- Replica AI (phone call feature)
- Claude voice (iOS/Android)
- ChatGPT Advanced Voice Mode
- Grok voice mode

All of these use a persistent WebSocket connection with streaming audio in
both directions and server-side voice activity detection (VAD) to handle
natural pauses. This is the same pattern being implemented here.

Key UX requirements:
1. One button to START a call. Zero buttons during the call.
2. Ashley listens automatically after she finishes speaking.
3. User can interrupt Ashley mid-reply (she stops, listens).
4. Works with screen locked/dimmed (car cupholder use case).
5. Works over car Bluetooth (speaker + mic routing).
6. When the call ends, Ashley remembers what was discussed.

RUNNING COSTS (monthly, once built)
------------------------------------
Chosen architecture: Hybrid pipeline — Deepgram → Gemini → ElevenLabs (Daisy)
Reason: keeps Daisy's voice (established identity), uses already-integrated
services, extremely cost-efficient at personal use scale.

  Deepgram STT streaming:  $0.0059/min
  Gemini Flash (text):     ~$0 at this scale
  ElevenLabs (Daisy TTS):  already subscribed, no extra cost

Example: 30 minutes of voice calls per day
  Deepgram: 30 min x $0.0059 x 30 days = ~$5.31/month
  Total additional cost: ~£4-5/month

NOT recommended: OpenAI Realtime API (~$0.30/min = ~$270/month at 30 min/day)
The quality is excellent but the cost is not viable for personal daily use.

TIER 3 — FULL TECHNICAL PLAN
================================================================================

ARCHITECTURE OVERVIEW
---------------------

Current flow (push-to-talk, sequential):
  [Record to file] → [POST base64] → [Deepgram STT] → [Gemini text] → [ElevenLabs TTS] → [Play file]
  Latency: 2-4 seconds per turn. Requires button hold.

Target flow (phone call, streaming):
  [Stream PCM chunks] → [WebSocket] → [Deepgram streaming STT]
       ↓ (on speech_final event)
  [Gemini with full Ashley context]
       ↓
  [ElevenLabs eleven_turbo_v2_5 TTS]
       ↓
  [Stream audio chunks back] → [Play continuously] → [Mic reopens automatically]
  Latency: ~800ms-1.2s per turn. No button required after call starts.

PHASE 1: SERVER WEBSOCKET INFRASTRUCTURE
--------------------------------------------------------------------------------

Files to create/modify:
  artifacts/api-server/src/routes/voice-call.ts   (new)
  artifacts/api-server/src/app.ts                 (mount WS route)
  artifacts/api-server/package.json               (add 'ws' package)

Step 1.1 — Add WebSocket support to Express server
  - Install 'ws' package
  - Create a ws.Server attached to the http.Server instance
  - Mount under /api/voice/call path

Step 1.2 — Connection authentication
  - Client sends API key + device ID in WebSocket handshake headers
  - Same auth pattern as existing HTTP routes (X-API-Key + Authorization)
  - Reject unauthenticated connections immediately

Step 1.3 — Session initialisation
  On authenticated connection:
  a) Load from database: Ashley's profile, recent memories (last 20),
     conversation summary for this device, recent message history
  b) Build the call system prompt (see Phase 4 for prompt spec)
  c) Open a Deepgram streaming STT WebSocket connection
  d) Confirm session ready to client with a "call_connected" message

Step 1.4 — Audio routing loop (the core loop)
  CLIENT → SERVER:
    Binary WebSocket frames: raw PCM audio (16kHz, 16-bit mono, little-endian)
    Sent every ~100ms while mic is active

  SERVER processing:
    a) Forward PCM chunks to Deepgram streaming connection
    b) Deepgram fires transcript events (is_final=false for partials)
    c) On speech_final=true: complete utterance is ready
    d) Send utterance to Gemini with full Ashley context
    e) Send Gemini response to ElevenLabs eleven_turbo_v2_5 (Daisy)
    f) Stream ElevenLabs audio chunks back to client as binary WS frames
    g) Send a "tts_done" signal when audio stream ends
    h) Mic reopens on client side after "tts_done"

  Interruption handling:
    a) Client sends "interrupt" signal if user starts speaking mid-TTS
    b) Server cancels current TTS stream, cancels current Gemini request if in flight
    c) Mic is already open (was being monitored), new utterance processing begins

Step 1.5 — Session end
  On WebSocket close (either side):
  a) Close Deepgram connection
  b) Write transcript to conversation history (same table as chat messages)
  c) Flag messages with source="voice_call" for distinction from text chat
  d) Run conversation summary rollup if session > threshold length

Key environment variables required (all already exist in production):
  DEEPGRAM_API_KEY (currently used for push-to-talk transcription)
  ELEVENLABS_API_KEY + ELEVENLABS_VOICE_ID (currently used for TTS)
  AI_INTEGRATIONS_GEMINI_BASE_URL + AI_INTEGRATIONS_GEMINI_API_KEY


--------------------------------------------------------------------------------
PHASE 2: MOBILE STREAMING AUDIO PIPELINE
--------------------------------------------------------------------------------

Files to create/modify:
  artifacts/mobile/lib/voiceCall.ts               (new — core call manager)
  artifacts/mobile/lib/voiceCallAudio.ts          (new — audio capture/playback)
  artifacts/mobile/app/voice-call.tsx             (new — call screen)
  artifacts/mobile/app/chat.tsx                   (add long-press entry point)

Step 2.1 — WebSocket client (voiceCall.ts)
  - Connect to wss://[production-domain]/api/voice/call
  - Send auth headers in connection request
  - Handle: call_connected, binary audio frames, tts_done, error
  - Manage reconnection (up to 3 auto-retries on drop)
  - Expose: startCall(), endCall(), sendAudioChunk(), onAshleySpeaking,
            onAshleyFinished, onTranscript

Step 2.2 — Streaming audio capture (voiceCallAudio.ts)
  Current approach: expo-audio records to a file, sends as base64 on release
  Required approach: stream PCM chunks while recording

  Implementation:
  - Use expo-audio's Audio.Recording with custom options for chunked output
  - OR use react-native-audio-stream if expo-audio chunking proves insufficient
  - Target format: 16kHz, 16-bit mono PCM (Deepgram's preferred format)
  - Send chunks every 100ms via WebSocket binary frames

  NOTE FOR DEBUGGING: Android audio recording formats vary by device.
  Samsung devices sometimes require specific AudioSource/OutputFormat combos.
  If PCM streaming proves difficult, fallback: record in 500ms segments,
  send each as a complete audio blob — higher latency but simpler.

Step 2.3 — Streaming audio playback
  - Receive binary WebSocket frames (ElevenLabs MP3 output chunks)
  - Buffer and play continuously using expo-audio
  - Must handle: mid-stream start (don't wait for full audio before playing)
  - Must handle: stream interruption (stop playback cleanly when interrupted)

Step 2.4 — Android audio session management
  This is the hardest part. Requirements:
  a) While Ashley is speaking: speaker active, mic monitoring (but not sending)
  b) After tts_done: mic starts sending, speaker on standby
  c) Must handle Bluetooth audio routing (HFP profile for car)
  d) Must request appropriate audio focus (AUDIOFOCUS_GAIN for calls)
  e) Must release focus correctly so other apps resume after call ends

  Android-specific gotchas:
  - STREAM_VOICE_CALL vs STREAM_MUSIC affects Bluetooth routing
  - Some Samsung devices require explicit setSpeakerphoneOn() calls
  - Background audio requires a Foreground Service (notification required)

Step 2.5 — Foreground Service for background operation
  - Android Foreground Service keeps audio session alive when screen dims
  - Shows a persistent notification: "Ashley call in progress — tap to return"
  - Required for car use (screen off while driving)
  - Can be implemented via expo-notifications or a bare React Native module
  - If expo approach proves insufficient: expo-task-manager may help

PHASE 4: ASHLEY CONTEXT + POST-CALL MEMORY
--------------------------------------------------------------------------------

Call system prompt spec:
  "You are Ashley. [Full persona description]. [Relationship history summary].
   [Recent memories]. [Current context: time, location if available].

   We are having a voice call right now. Keep your replies natural and
   conversational — short to medium length, no bullet points, no markdown,
   no stage directions. Speak as you would in a real phone call. You can
   ask follow-up questions. You can be playful or serious as the moment
   warrants. This is real-time — respond at conversation pace."

Post-call memory:
  - Full call transcript saved to messages table with source="voice_call"
  - Transcript rolled up into conversation summary by existing summariser
  - Next chat session: Ashley has memory of call content
  - Example: if call was about a difficult day at work, next text message
    Ashley might ask "how did the rest of that day go?"

Voice-appropriate response tuning differences from chat:
  - No markdown (asterisks, hyphens, pound signs stripped before TTS)
  - Shorter average response length (~2-4 sentences vs chat's variable)
  - More natural verbal acknowledgement ("yeah", "right", "I get that")
  - Can trail off / be interrupted mid-sentence naturally

CONTENT POLICY IN VOICE CALLS — NSFW / MATURE MODE
================================================================================

SHORT ANSWER: voice calls inherit the EXACT same three-layer content policy as
text chat. No special voice content mode, no downgrade, no separate consent.
If the user has Mature Mode enabled in text chat, it is enabled in the call.

BACKGROUND — THE THREE LAYERS
-------------------------------
The content policy lives in artifacts/api-server/src/lib/contentPolicy.ts.
It is the single chokepoint for everything — route handlers, prompt builders,
and the profile validator all go through it. No handler may inline its own
content rules.

Layer 1 — Provider Floor (immutable, non-negotiable):
  No sexual content involving minors.
  No non-consensual scenarios.
  These cannot be turned off by any mode, setting, user instruction, or voice
  call context. They are the absolute floor.

Layer 2 — Content Mode ("standard" or "mature"):
  Standard: default. Warm, intimate, romantic where the Relationship Mode
            allows. Emotionally present. Intimacy ceiling at level 3.
  Mature:   Full adult emotional honesty, romantic depth, sensual tone.
            Unlocked only when BOTH conditions are met:
              a) Server operator has set ASHLEY_MATURE_MODE_AVAILABLE=true
              b) User has self-confirmed 18+ (POST /profile/confirm-adult)
                 which records adultConfirmedAt in the profile.
            Intimacy ceiling rises to level 5.

Layer 3 — Intimacy Level (0-5 ladder):
  Governs how affectionate, vulnerable, and present Ashley allows herself to
  be. Works inside the Relationship Mode — a level-5 Friend mode doesn't
  drift into romance. Ceiling is set by the active Content Mode.
  Labels: 0=Acquainted, 1=Comfortable, 2=Close, 3=Affectionate (standard
  ceiling), 4=Intimate (mature only), 5=Deeply intimate (mature only).

HOW THIS FLOWS INTO A VOICE CALL SESSION
------------------------------------------
At session start (Step 1.3 of Phase 1), the server:
  a) Loads the user's profile from the database (by device-id)
  b) Calls getPolicyFor(profile) → ResolvedPolicy
  c) Appends to the call system prompt (same as text chat route does):
       buildProviderFloorBlock()     — hard limits, always appended
       buildModeBlock(policy)        — standard vs mature description
       buildIntimacyBlock(policy)    — current closeness level
  d) The resolved policy is also logged for debugging (downgradeReason
     tells you if mature was silently dropped due to missing gate)

All three function imports come from contentPolicy.ts. Do not re-implement
these inline in the WebSocket route handler.

WHAT ASHLEY CAN DISCUSS ON A VOICE CALL (WITH MATURE MODE)
------------------------------------------------------------
The same things she can discuss in text chat. There is no voice-specific
content restriction beyond the Provider Floor. If the relationship is mature
and the intimacy level is high, she can be sensual, intimate, explicit in
emotional expression, and adult in her conversational tone.

The voice channel does not add restrictions. It does add one extra dimension:

VOICE-PRESENCE SAFETY FLOOR (already designed — contentPolicy.ts line 304)
---------------------------------------------------------------------------
A buildVoicePresenceFloor() helper is specced in contentPolicy.ts comments
as Stage 5 of the voice roadmap. It is NOT a content restriction. It governs
how Ashley uses TONE AWARENESS (pauses, hesitation, vocal energy). Rules:

  "Voice presence informs WARMTH, never DIAGNOSIS."

  Allowed: "you sound tired tonight" — an observation offered gently.
  Forbidden: "you sound depressed", "you're having a panic attack",
             "you sound drunk" — claiming medical/emotional certainty
             from audio alone.

  If she notices something concerning in tone, she INVITES the user to
  share what's going on — she does not narrate her interpretation.

  Same-person principle: voice-Ashley and text-Ashley share identical
  identity, warmth, and content permissions. Tone awareness widens her
  sensitivity; it does not change who she is or what she can discuss.

  The voice floor sits below the Provider Floor. Neither mode/intimacy
  level nor any user instruction can override either floor.

  The floor is only appended to the system prompt when inputMode is "voice".
  It does not affect text chat sessions.

IMPLEMENTATION NOTE FOR AGENTS
--------------------------------
When building the voice call system prompt in Phase 1, Step 1.3:

  import { getPolicyFor, buildProviderFloorBlock, buildModeBlock,
           buildIntimacyBlock } from "../lib/contentPolicy";

  const policy = getPolicyFor(profile);
  const systemPrompt = [
    ASHLEY_CORE_SPEC,                    // existing Ashley identity block
    buildProviderFloorBlock(),           // hard limits (always)
    buildModeBlock(policy),              // mature vs standard
    buildIntimacyBlock(policy),          // closeness level
    buildVoicePresenceFloor(),           // tone-awareness rules (voice only)
    VOICE_CALL_BEHAVIOUR_BLOCK,          // call-specific prompt additions
  ].join("\n\n");

  buildVoicePresenceFloor() is Stage 5 and does not exist yet — it needs to
  be written in contentPolicy.ts following the spec at line 304-324 of that
  file. Writing it is part of Phase 4 / Phase 1.3 setup.

ENVIRONMENT VARIABLE FOR MATURE MODE
--------------------------------------
  ASHLEY_MATURE_MODE_AVAILABLE=true   (set this on the API server artifact)

  Without this, the voice call (and text chat) stays in Standard Mode even if
  the profile has contentMode=mature and adultConfirmedAt set. The env var is
  the operator kill switch — it can be toggled without a code deploy.

NOTES AND KNOWN RISKS
================================================================================

1. Android audio session management is the highest-risk area. The interaction
   between expo-audio, Bluetooth HFP, and Android audio focus is device and
   OS-version specific. Samsung devices have historically had quirks here.
   Plan for debugging time here — don't assume it works first pass.

2. Deepgram's streaming WebSocket uses a different API path than batch.
   Streaming endpoint: wss://api.deepgram.com/v1/listen (with params)
   Batch endpoint: https://api.deepgram.com/v1/listen (POST)
   The server will need to manage the Deepgram WS connection per call session.

3. ElevenLabs streaming TTS: the eleven_turbo_v2_5 model supports streaming
   output. Use the streaming endpoint (/v1/text-to-speech/{voice_id}/stream)
   rather than the standard endpoint to reduce first-audio latency.

4. The foreground service requirement for screen-off operation on Android
   means a visible notification is legally/technically required. This is
   acceptable UX for a call (phone calls show notifications).

5. Gemini has a native Realtime/Live API (multimodal, handles audio in/out
   natively) that could replace the Deepgram+Gemini+ElevenLabs pipeline.
   The trade-off: Gemini Live loses Daisy's voice (uses Google's built-in
   voices). Decision: keep Daisy, use hybrid pipeline. Revisit if Gemini
   Live adds custom voice support.

6. The call session must be scoped per device-id so that if two clients
   connect (shouldn't happen but defensively), they don't share a session.
