--------------------------------------------------------------------------------
PROACTIVE ISSUES
--------------------------------------------------------------------------------

1. ttsCompleteRef is initialized to true at line 160 (just below the fix):
     const ttsCompleteRef = useRef(true);
   This ref is the DEVICE playback completion guard. Initializing to true means
   the device is considered "done playing" before any audio has played. This is
   governed by the call_connected handler (line 537), which opens the mic
   immediately on connection — the initial true is intentional there (no audio
   to play before the first turn). The fix to ttsServerDoneRef is the relevant
   change; ttsCompleteRef's initialization is consistent with its existing logic.
   Noting it here because a future reader might ask the same question.

2. The full-cycle echo loop (once started, sustaining itself) is not addressed
   by this fix. Case 5 in the drain test shows the sustaining path
   (srvDone=true, responseEnd=false after speech_start reset) still takes
   OPEN_MIC. This fix removes the condition that caused the FIRST premature
   mic-open (the initialization defect). If the echo loop has a second entry
   point, this fix does not close it. Kane's soak test will determine whether
   a second entry point exists.
