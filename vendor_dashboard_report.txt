2. VENDORS TAB DATA SOURCE — CONFIRMED, AND ONE REAL BUG FOUND + FIXED
====================================================================

Confirmed: the Base44 Vendors tab is a no-code frontend that calls this
repo's API directly — GET /dashboard/vendors (list) and
GET /dashboard/vendors/:id (detail). Both read straight from the local
SQLite `vendors` table via getDb(). There is no caching layer and no other
data source in between — whatever is in that table is what the tab shows,
live, on every load.

The bug: the ethics_check_verdict / ethics_check_reasoning / ethics_check_date
/ ethics_check_activity_log_id columns were being written correctly by
runFormalVendorEthicsCheck, but the API response mapper (rowToVendor) never
included them. So even a vendor that WAS correctly re-checked would show the
result nowhere on the Vendors tab — you'd only see the older, generic
`ethicalVerificationStatus`/`forcedLabourCheckStatus` fields, not the actual
verdict/reasoning/decision-date from the formal check. This is very likely
why the two vendors "looked stale" even before I re-ran anything: the tab
was never wired to display the field that proves a formal check happened.

Fixed and deployed: rowToVendor now also returns adverseMediaCheckStatus/
Notes/Date and ethicsCheckVerdict/Reasoning/Date/ActivityLogId. Confirmed
live in the GET /dashboard/vendors response above — both fields are now
populated for Gildan and HanesBrands.

Also fixed in the same pass: the permanent-disqualification code path in
runFormalVendorEthicsCheck wrote the activity_log row AFTER already running
the vendor UPDATE, so ethics_check_activity_log_id was always left NULL.
Reordered so the log write happens first and its id is persisted on the
vendor row — confirmed non-null (175/176) after re-running both checks.

3. DELETE / ARCHIVE FROM THE VENDORS TAB
====================================================================

Backend now supports both, live:

- DELETE /dashboard/vendors/:id  (already existed, unchanged)
    Permanently destroys the row. PIN-gated: body { "pin": "<KANE_APPROVAL_PIN>" }.
    Irreversible — use for genuine mistakes/duplicates only.

- POST /dashboard/vendors/:id/archive  (NEW)
    Soft-delete: sets archived_at, row and full ethics-check history survive.
    PIN-gated the same way: body { "pin": "<KANE_APPROVAL_PIN>" }.
    Archived vendors are excluded from GET /dashboard/vendors by default.

- POST /dashboard/vendors/:id/unarchive  (NEW)
    Restores an archived vendor back into the default list. Same PIN gate.

- GET /dashboard/vendors?includeArchived=true  (NEW query param)
    Shows archived vendors too, for an "Archived" view if you ever want one.

Smoke-tested live (no PIN available to me, so I couldn't run a full
archive/restore cycle end-to-end, but confirmed the routes are registered,
reachable, and correctly PIN-gated — both new endpoints return HTTP 403
"Invalid or missing approval PIN" rather than 404, and the
?includeArchived=true param is accepted without error):

  POST /vendors/<id>/archive    (wrong pin) -> 403 Invalid or missing approval PIN
  POST /vendors/<id>/unarchive  (wrong pin) -> 403 Invalid or missing approval PIN
  GET  /vendors?includeArchived=true        -> 200, count: 2 (as expected, none archived)

IMPORTANT — the actual button doesn't exist yet: Base44 is a separate
no-code editor, not code in this git repo, so I cannot add UI elements to
the Vendors tab myself. The backend is ready; wiring a "Delete" and/or
"Archive" button into the Base44 card view is a Base44-editor task on your
end. Exact call shape for a button:

    POST https://michael-orchestrator-production.up.railway.app/dashboard/vendors/{id}/archive
    headers: x-api-key: <MICHAEL_API_KEY>, Content-Type: application/json
    body: { "pin": "<your KANE_APPROVAL_PIN>" }

  Delete button (permanent):
    DELETE https://michael-orchestrator-production.up.railway.app/dashboard/vendors/{id}
    same headers/body shape as above
