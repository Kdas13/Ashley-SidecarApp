ITEM 1 — GENERIC APPROVAL REQUEST SYSTEM
────────────────────────────────────────────────────────────────────────────────

All 11 gate types now route through one queue. Endpoints:

  GET   /dashboard/approval-requests
        List all. Filter: ?gate_type=gate-vendor&status=pending&projectId=nrnl

  GET   /dashboard/approval-requests/:id
        Single request.

  POST  /dashboard/approval-requests
        Create. Body: { projectId, gateId, summary, payload }
        Validates gateId against constitution. Returns 400 for unknown gate.

  POST  /dashboard/approval-requests/:id/approve
        Approve. Optional body: { note }

  POST  /dashboard/approval-requests/:id/approve-with-edits
        Approve with edit instructions. Required body: { editsNote }

  POST  /dashboard/approval-requests/:id/reject
        Reject. Optional body: { note }

Status values:
  pending / approved / approved_with_edits / rejected / notified

"notified" is used for autonomous-pause notifications (ads-3) — visible on
the dashboard but does not require Kane action to resolve.

Every row carries:
  constitutionVersion — the version active when the request was created
  gateId              — one of the 11 valid values below
  summary             — human-readable one-liner
  payload             — JSON, type-specific per gate

Valid gate IDs (all 11):
  gate-vendor            Vendor selection (final pick)
  gate-design            Design sign-off (all generated images)
  gate-pricing           Pricing / margin / sell price before go-live
  gate-ads               Ad spend and creative (per-campaign)
  gate-payment-account   Bank/payment account setup
  gate-comms             Brand-representing outbound comms
  gate-email             Per-draft email approval
  gate-vendor-non-ukeu   Non-UK/EU vendor (fallback tier)
  gate-new-channel       New marketing channel/platform
  gate-customer-service  Refund / replacement / policy exception
  gate-quota             Search quota at 90% of monthly cap (NEW in v6)

ITEM 3 — AD CAMPAIGN ENTITY
────────────────────────────────────────────────────────────────────────────────

New columns added to ad_campaigns:
  approved_budget_cap     The fixed cap approved via gate-ads
  current_spend           Actual spend to date
  ad_set_allocations      JSON — reallocation within cap
  approval_request_id     Links to the gate-ads ApprovalRequest
  pause_reason            Required for autonomous pause (ads-3)
  paused_at               Timestamp of pause
  ended_at                Timestamp of end
  approved_at             Timestamp of Kane approval
  platform_campaign_id    External platform reference (Meta etc.)

Endpoints:

  GET   /dashboard/campaigns
        List. Filter: ?projectId=&status=

  GET   /dashboard/campaigns/:id
        Single campaign, full field set.

  POST  /dashboard/campaigns
        Create draft. Body: { projectId, platform, audienceSegment,
        targetCollection, creativeAssetUrls }
        Status starts as "draft". Gate-ads ApprovalRequest required before launch.

  POST  /dashboard/campaigns/:id/increase-cap
  PATCH /dashboard/campaigns/:id/increase-cap
        HARD BLOCKED — always returns 403.
        Response includes constitutionRule: "ads-2" and constitutionVersion.
        This is enforced at the request level, not advisory.

  PATCH /dashboard/campaigns/:id/reallocate
        Reallocate ad sets within approved cap (ads-1).
        Body: { allocations: { "adSetName": amount } }
        Enforced: sum of allocations must not exceed approved_budget_cap.
        Returns 400 with exact numbers if it would breach.

  PATCH /dashboard/campaigns/:id/pause
        Autonomous pause (ads-3). Allowed without approval.
        Body: { projectId, pauseReason }
        pauseReason is required — returns 400 without it.
        Routes through runConstitutionalAction: action is logged to
        activity_log with constitution_version. Also creates a "notified"
        ApprovalRequest so it surfaces on the dashboard.

  PATCH /dashboard/campaigns/:id/resume
        BLOCKED — returns 403. Instructs caller to submit a gate-ads
        ApprovalRequest with payload { campaignId, action: "resume" }.
        Constitution rule: ads-4.

Spend-control summary:
  ads-1  Reallocation within cap: allowed, enforced numerically
  ads-2  Cap increase: blocked at API layer (403), no workaround possible
  ads-3  Autonomous pause: allowed, requires reason, auto-logs + notifies
  ads-4  Resume / launch / cross-campaign budget move: blocked, gate required

ITEM 5 — ACTIVITY LOG CONSTITUTION VERSION — CONFIRMED ALREADY PRESENT
────────────────────────────────────────────────────────────────────────────────

No code change needed. Documenting for the audit trail.

activity_log.constitution_version TEXT NOT NULL has been in the schema
since the first commit. runConstitutionalAction() stamps it on every write.
rowToDecision() exposes it as constitutionVersion on every response from
GET /dashboard/decisions and GET /dashboard/decisions/:id.

Every action that touches the LLM or mutates state — vendor ethics check,
email approval, image generation, supplier search, general chat reply,
campaign pause — goes through runConstitutionalAction. The version recorded
is the version that was live at the moment of execution, not the current
version at time of read. This is the mechanism described in Section 2:
"diagnose as rule-was-wrong vs rule-wasn't-followed."

The field is already live in production.
