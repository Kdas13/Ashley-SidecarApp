ANSWER TO ISSUE 1 QUESTION ("is there an energy/VAD threshold or does ANY
mic-open + brief audio commit auto-submit regardless of content?"):

TWO separate paths to submission:

PATH A — metering-based VAD (lines 421–450):
  Requires: recorder.metering non-null AND db > SILENCE_DB (-45 dBFS) AND
  ≥500ms of continuous sound AND ≥3500ms of silence after that sound.
  Only then calls submitSegment.

PATH B — 4000ms fallback timer (lines 493–503):
  Fires unconditionally 4s after every openMic call.
  Requires only: phaseRef === "listening" or "user_speaking".
  Since isUserSpeakingRef stays false when metering is null, phase stays
  "listening" — the timer always fires and always calls submitSegment.

submitSegment itself (lines 351–413):
  The ONLY guard against sending garbage audio is:
    (a) recorder.stop() returning null (no audio object) → reopen mic
    (b) STT returning an empty transcript → reopen mic (starts new 4s timer)
  There is NO amplitude threshold, duration threshold, or content filter in
  submitSegment. Any non-empty STT result is sent as speech_final.

CYCLE MECHANISM (what Kane observed):
  openMic() → 4s timer starts → timer fires → submitSegment() → STT (empty
  because AirPods mic is recording near-silence) → reopen mic → new 4s timer
  → repeat every ~5–6s indefinitely.

  AirPods change: with AirPods the STT consistently returns empty (no text for
  near-silence) so the cycle produces no chat messages. Without AirPods, room
  audio or Ashley's speaker bleed may produce phantom STT text.

  LLM token. This is a network call to the Gemini API (forceProvider: "gemini").
  If Gemini is slow, the loop just waits.

"Is there a budget/timeout on the LLM call itself, separate from MAX_TURN_BUDGET_MS?"
  There are TWO timers, both started AFTER speech_start is sent:

  FIRST_TOKEN_WATCHDOG_MS = 8,000ms:
    If no first token within 8s, plays the "Hang on, I'm thinking..." auxiliary
    clip. This does NOT terminate the LLM call — it just plays a filler clip.
    The LLM call continues waiting. The watchdog is cancelled on first token.

  MAX_TURN_BUDGET_MS = 45,000ms:
    If the full turn (including TTS) hasn't completed within 45s, the hard budget
    fires, cancels the LLM stream via AbortController, and sends tts_done(main)
    to the client. This is the ONLY automatic recovery from a hung LLM call.

  There is NO timeout on VoiceContextAssembler.assemble() (Stage 5, before
  speech_start). If the DB query hangs, speech_start is never sent, the hard
  budget is never armed, and nothing is ever sent to the client. In that case
  the client would stay in "thinking..." phase with no recovery.

  In Kane's Issue 2 log, speech_start(main) WAS received, so the context
  assembly succeeded. The freeze is during the LLM streaming phase (Stage 6).

"If the LLM call hangs or errors silently, does anything get sent back?"
  Three failure paths, in code order:
    (a) AbortError / APIUserAbortError: catch block exits silently.
        No message sent to client. The finally block runs.
        Then the response_end block runs — but only if session.currentSpeechId === speechId.
        If the abort came from the hard budget (which also sets currentSpeechId=null at line 509),
        response_end is skipped. The hard budget already sent tts_done directly.
        Net: client gets tts_done(main) from hard budget. Lifecycle resumes.

    (b) Non-abort error (Gemini HTTP error, network error, etc.):
        speakClipOrText(kind="main") plays "say-that-again" clip.
        Client gets speech_start(main)+tts_done(main) without response_end.
        Mic opens via srvDone=true, responseEnd=false path.
        NOTE: this also calls the response_end block immediately after,
        but session.currentSpeechId was NOT changed by speakClipOrText, so
        response_end IS sent. Both tts_done (from speakClipOrText) and then
        response_end arrive at the client. Client already opened mic at tts_done.

    (c) LLM stream hangs forever (no error, no abort):
        streamChatText's for-await loop waits forever.
        Hard budget fires at 45,000ms → sends tts_done(main) → lifecycle resumes.
        Between speech_start and 45s: client shows "Thinking..." with no feedback
        except the 8s thinking clip (auxiliary, doesn't change phase).

  If Kane's freeze lasted a visible time but recovered without user action after
  ~45s: scenario (c). If it never recovered: either the WS connection dropped
  during the 45s window, or session.currentSpeechId was null when the hard
  budget fired (gate check at line 505 failed) → tts_done was not sent.

ANSWER: there is NO client-side timeout for the thinking phase. Once the client
enters phase="thinking" (after sending speech_final at line 407), it has no timer,
no watchdog, and no recovery mechanism of its own. The client stays in "thinking"
phase until a WS message from the server changes its state:
  - speech_start(main) → transitions to "speaking" (or stays in "thinking" until
    audio queue starts; the phase is set to "speaking" inside playNext when a
    track starts playing at line 249)
  - tts_done(main) arriving without prior speech_start (safety path) → sets
    ttsServerDoneRef=true, triggers drain, mic opens, phase → "listening"
  - WS close (onclose handler at lines 677–699) → either reconnect or "ended"

If the server hangs for 45s and then sends tts_done via the hard budget, the
client recovers. If the WS connection drops during that window and reconnect
fails, the client reaches "ended" state. If neither happens (hang with open WS,
hard budget gate failing), the client is stuck in "thinking" indefinitely.

There is no UI timeout that would show an error after N seconds of thinking.
