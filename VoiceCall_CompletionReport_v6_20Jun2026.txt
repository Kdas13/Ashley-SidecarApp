FINDING 2 — INTERRUPT HANDLER PLAYBACK GATE FIX (Option A)
================================================================================

FILE: artifacts/api-server/src/index.ts (interrupt case)

WHAT CHANGED:

Three additions to the interrupt handler, between cancelCurrentTurn and the
interrupt_ack send:

1. CLEAR THE PLAYBACK GATE:

  if (currentSession.playbackConfirmTimeout) {
    clearTimeout(currentSession.playbackConfirmTimeout);
    currentSession.playbackConfirmTimeout = null;
  }
  currentSession.awaitingPlaybackConfirm = false;
  currentSession.responseComplete = false;

2. CAPTURE AND CLEAR THE PENDING UTTERANCE (before interrupt_ack):

  const drainUtterance   = currentSession.pendingUtterance;
  const drainUtteranceId = currentSession.pendingUtteranceId;
  currentSession.pendingUtterance   = null;
  currentSession.pendingUtteranceId = null;

3. DRAIN (after interrupt_ack is sent):

  if (drainUtterance && drainUtteranceId) {
    logger.info(..., "voice: interrupt — draining queued utterance");
    void VoiceOrchestrationService.handleSpeechFinal(
      currentSession, drainUtterance, drainUtteranceId,
    ).catch((err) =>
      logger.error({ err, deviceId }, "voice: interrupt drain failed"),
    );
  }

WHY THE ORDER MATTERS:

  cancelCurrentTurn first  — clears currentSpeechId so in-flight TTS sees null
  gate clear second        — clears awaitingPlaybackConfirm before interrupt_ack
  interrupt_ack third      — client transitions to listening state
  drain fourth (void/async)— LLM starts only after client is in listening state

WHAT THIS FIXES IN THE LOG SEQUENCE (from the 59s freeze):

  Before fix:
    T+9.8s  interrupt handled
            awaitingPlaybackConfirm: still true
    T+15.1s "Sorry." speech_final → queued — awaiting playback_confirmed
    T+~23s  15s safety timeout fires, guard fails silently (currentSpeechId=null)
    T+74.4s call_end — "Sorry." was never processed

  After fix:
    T+9.8s  interrupt handled
            awaitingPlaybackConfirm = false  ← CLEARED
            drainUtterance captured
            interrupt_ack sent
            handleSpeechFinal("Sorry.") starts immediately
    T+~10s  LLM reply begins — call recovers in < 1 second

WHAT THIS DOES NOT FIX:

  The 15s safety timeout's guard (`currentSpeechId === speechId`) remains
  vulnerable to the same failure mode from any other path that clears
  currentSpeechId before the timeout fires. Option B (drop the speechId check
  from the guard) should be done as a follow-up hardening pass. Today it is
  belt-and-suspenders; after this fix it is no longer the last line of defence.
