WHAT CHANGED AND WHY (protocol summary)
================================================================================

Old protocol (broken):
  speech_start → chunks → sentence_end* → tts_done → [client sends playback_complete]
  → server sends tts_done → client opens mic

  Bug: client sent playback_complete whenever queue drained, including
  between sentences. Inter-sentence drain was indistinguishable from
  end-of-response drain.

New protocol (this brief):
  speech_start → chunks → sentence_end* → response_end → [client sends playback_confirmed]
  → server sends tts_done → client opens mic

  Fix: response_end is a server-authoritative signal meaning "no more audio
  for this turn". Client only sends playback_confirmed after receiving both
  response_end AND observing a queue drain. Inter-sentence drains are
  silently ignored (new log line: "waiting for next sentence").

All audio paths now follow the same lifecycle:
  LLM responses:  runLLMAndTTSPipeline() → response_end
  Silence check:  speakFallback() → response_end
  Farewell:       speakFarewell() → speakFallback() → response_end → session finalised
